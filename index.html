Introduction to Operating Systems

What an operating system is, what goals it serves, the four components of a computer system, and the many roles the term "OS" covers — from toasters to spacecraft.

What is an Operating System?

An operating system is a program that acts as an intermediary between a user of a computer and the computer hardware.

The Three Goals of an OS Execute user programs and make solving user problems easier · Make the computer convenient to use · Use the hardware efficiently.

Computer System Structure — Four Components

A computer system can be divided into four components, each layered on the one below it:

①

Hardware

Provides the basic computing resources — CPU, memory, and I/O devices.

②

Operating System

Controls and coordinates the use of hardware among various applications and users.

③

Application Programs

Define the ways system resources solve users' problems — word processors, compilers, browsers, databases, games.

④

Users

People, machines, and other computers that use the system.

What Operating Systems Do — A Matter of Viewpoint

What an OS does depends on the point of view:

  • Users want convenience, ease of use and good performance — they don't care about resource utilization.
  • Shared computers (mainframe/minicomputer) must keep all users happy — here the OS is a resource allocator and control program.
  • Dedicated systems (workstations) have dedicated resources but use shared resources from servers.
  • Mobile devices are resource-poor — optimized for usability and battery life, using touch screens and voice.
  • Embedded computers (devices, automobiles) have little or no user interface and run without intervention.

Defining the Operating System

No universally accepted definition

A good approximation: "everything a vendor ships when you order an operating system" — but this varies wildly.

K

Kernel

The one program running at all times on the computer — the core of the OS.

SP

System Program

Ships with the OS, but is not part of the kernel.

AP

Application Program

All programs not associated with the operating system.

Modern general-purpose and mobile OSes also include middleware — a set of software frameworks providing additional services (databases, multimedia, graphics) to application developers.

Where OSes Live Operating systems are present in toasters through ships, spacecraft, game machines, TVs and industrial control systems. The OS was born when fixed-use military computers became general-purpose and needed resource management and program control.

Computer-System Structure & Operations

How the hardware is organised, the central role of interrupts, the storage hierarchy, multiprocessor architectures, bootstrapping, multiprogramming vs multitasking, and dual-mode operation.

Computer-System Organization

One or more CPUs and device controllers connect through a common bus providing access to shared memory, executing concurrently and competing for memory cycles.

  • Each device controller is in charge of a particular device type and has a local buffer.
  • Each controller type has an OS device driver to manage it.
  • The CPU moves data between main memory and local buffers; I/O is from the device to the controller's local buffer.
  • The controller informs the CPU it has finished by causing an interrupt.

Interrupts — The Heart of the OS

An operating system is interrupt-driven An interrupt transfers control to the interrupt service routine via the interrupt vector (a table of service-routine addresses). The architecture must save the address of the interrupted instruction.

Interrupt Handling

  • The OS preserves the CPU state by storing the registers and program counter.
  • It determines which type of interrupt occurred.
  • Separate segments of code decide the action for each type.

Traps & Exceptions

A trap (or exception) is a software-generated interrupt caused either by an error (e.g. division by zero) or a user request (a system call).

I/O Structure & DMA

There are two methods for handling I/O after it starts:

Synchronous (blocking)

  • Control returns to the user program only upon I/O completion.
  • A wait instruction idles the CPU until the next interrupt.
  • At most one I/O request outstanding at a time — no simultaneous processing.
vs

Asynchronous

  • Control returns without waiting for I/O completion.
  • A system call lets the user wait for completion if desired.
  • A device-status table holds each device's type, address and state.
Direct Memory Access (DMA)

Used for high-speed I/O devices able to transmit near memory speed. The device controller transfers blocks of data directly from its buffer to main memory without CPU intervention — generating only one interrupt per block instead of one per byte.

Storage Hierarchy

Storage systems are organised in a hierarchy by speed, cost and volatility. Caching — copying information into faster storage — appears at every level; main memory can be viewed as a cache for secondary storage.

▲

Main Memory

The only large storage the CPU can access directly. Random-access, typically volatile DRAM.

HDD

Hard Disk Drives

Rigid platters with magnetic recording material; surface divided into tracks subdivided into sectors.

NVM

Non-Volatile Memory

Faster than hard disks and nonvolatile — increasingly popular as capacity rises and price drops.

Storage units The basic unit is the bit (0 or 1). A byte = 8 bits. A word is the architecture's native data unit (e.g. 8 bytes on a 64-bit machine). KB = 1,024 bytes; MB = 1,024² ; GB = 1,024³ ; TB = 1,024⁴ ; PB = 1,024⁵. Networking is the exception — measured in bits.

Computer-System Architecture

Multiprocessor (Parallel) Systems

Also called tightly-coupled systems. Advantages:

  • Increased throughput
  • Economy of scale
  • Increased reliability — graceful degradation / fault tolerance

Clustered Systems

Multiple systems working together, usually sharing storage via a SAN, providing a high-availability service:

  • Asymmetric — one machine in hot-standby mode.
  • Symmetric — multiple nodes running and monitoring each other.
  • Some are for HPC; some use a distributed lock manager (DLM).

Operating-System Operations

Computer Startup → Operation
1
Bootstrap ProgramLoaded at power-up from ROM/EPROM (firmware); initializes the system and loads the kernel.
2
Kernel LoadsStarts running, the one program present at all times.
3
System Daemons StartServices provided outside the kernel begin running.
4
Interrupt-Driven OperationThe kernel waits and responds to hardware and software interrupts (traps).

Multiprogramming vs Multitasking

Multiprogramming (Batch)

  • A single user cannot always keep the CPU and I/O devices busy.
  • Organizes jobs so the CPU always has one to execute.
  • A subset of jobs is kept in memory; one is run via job scheduling.
  • When a job must wait (e.g. for I/O), the OS switches to another job.
→

Multitasking (Timesharing)

  • The CPU switches jobs so frequently that users can interact with each — response time < 1 second.
  • Several ready jobs ⟶ CPU scheduling.
  • If processes don't fit in memory, swapping moves them in and out.
  • Virtual memory allows execution of processes not completely in memory.

Dual-Mode Operation

Exam Tip — User Mode vs Kernel Mode A hardware mode bit distinguishes user mode from kernel mode, letting the OS protect itself. A system call changes the mode to kernel; the return resets it to user. Privileged instructions are executable only in kernel mode — so users cannot set the mode bit to "kernel" themselves.

Operating System Services & System Calls

The services an OS provides to users and to itself, the user interfaces (CLI, GUI, touch), how system calls work, parameter passing, the types of system calls, and OS structures from monolithic to microkernel.

Services Helpful to the User

UI

User Interface

CLI, GUI, touch-screen, or batch — almost every OS has one.

PE

Program Execution

Load a program into memory and run it, ending normally or abnormally.

I/O

I/O Operations

A running program may need I/O involving a file or a device.

FS

File-System Manipulation

Read/write/create/delete/search files & directories, manage permissions.

C

Communications

Processes exchange information via shared memory or message passing.

ED

Error Detection

Constantly aware of errors in the CPU, memory, devices or user programs.

Services for Efficient System Operation

RA

Resource Allocation

Allocate CPU cycles, memory, file storage and devices among concurrent jobs.

L

Logging

Track which users use how much of which resources.

PS

Protection & Security

Control access to resources; authenticate users; defend against outsiders.

User Interfaces

⌨

Command-Line (CLI)

Allows direct command entry; implemented in the kernel or by a systems program. Multiple flavours called shells fetch a command and execute it.

🖱

GUI

User-friendly desktop metaphor with icons, mouse and folders — invented at Xerox PARC. Windows is GUI + CLI; macOS is "Aqua" over a UNIX kernel.

☝

Touchscreen

Gesture-based actions and a virtual keyboard; voice commands. No mouse.

System Calls

Definition

A system call is the programming interface to the services provided by the OS. Typically written in C/C++, they are mostly accessed by programs via a high-level API rather than directly.

The three most common APIs are Win32 API (Windows), POSIX API (UNIX, Linux, macOS), and the Java API (for the JVM).

How a System Call Works

  • A number is associated with each system call; the interface maintains a table indexed by these numbers.
  • The interface invokes the intended call in the kernel and returns status and any values.
  • The caller need know nothing about the implementation — just the API. Details are hidden by the run-time support library.

For example, a C program's printf() library call internally invokes the write() system call.

Parameter Passing — Three Methods

①

In Registers

Simplest — but there may be more parameters than registers.

②

In a Block / Table

Parameters stored in memory; the block's address is passed in a register (used by Linux & Solaris).

③

On the Stack

Parameters pushed onto the stack by the program and popped by the OS.

UNIX/Linux Process-Control System Calls

fork

fork()

Creates a new child process as an (almost) exact duplicate of the parent — copies of stack, data, heap and text.

exec

exec()

Replaces the process image with a new program.

wait

wait()

Synchronizes parent and child — the parent waits for the child.

exit

exit()

Terminates the process.

Why fork() is useful A network server can listen for client requests and fork a child to handle each one while it continues listening — simplifying design and permitting greater concurrency.

Operating-System Structure

Monolithic (Original UNIX)

Limited structuring forced by hardware. Two separable parts — system programs and the kernel (everything below the system-call interface and above the hardware): file system, CPU scheduling, memory management and more, all in one level.

Layered Approach

The OS is divided into layers built on top of one another — layer 0 is hardware, layer N is the user interface. Each layer uses only the operations and services of lower layers.

Microkernel (Mach)

  • Moves as much as possible from the kernel into user space.
  • Communication via message passing.
  • Benefits: easier to extend & port, more reliable and secure (less code in kernel mode).
  • Detriment: performance overhead of user↔kernel communication.
vs

Modules & Hybrid

  • Loadable kernel modules (LKMs) — object-oriented, each loadable as needed (Linux, Solaris).
  • Most modern OSes are hybrid: Linux & Solaris are monolithic + modular; Windows is mostly monolithic + microkernel; macOS is hybrid/layered over a Mach + BSD core.

Processes & File I/O System Calls

What a process is, the activities of process management, and the four low-level file I/O system calls — open(), read(), write() and close() — built on file descriptors.

The Process

Definition

A process is a program in execution — a unit of work within the system. A program is a passive entity; a process is an active entity. It needs resources (CPU, memory, I/O, files) to do its task, and termination requires reclaiming reusable resources.

Single- vs Multi-threaded

  • A single-threaded process has one program counter, executing instructions sequentially until completion.
  • A multi-threaded process has one program counter per thread.

Process Management Activities

  • Creating & deleting user and system processes.
  • Suspending & resuming processes.
  • Mechanisms for synchronization, communication and deadlock handling.

File I/O System Calls

open(), read(), write() and close() are low-level system calls used to interact with files through file descriptors, defined in unistd.h and fcntl.h.

File Descriptor (fd) A file descriptor is a small integer identifying an open file. Each process has its own descriptor table. Standard descriptors: 0 = stdin (keyboard), 1 = stdout (console), 2 = stderr.
CallSyntaxPurpose & Return
open()int fd = open(pathname, flags, mode)Open/create a file and obtain a descriptor. Returns ≥ 0 on success, −1 on failure.
read()ssize_t read(fd, buf, count)Read bytes into a buffer. Returns bytes read (>0), 0 at EOF, −1 on error.
write()ssize_t write(fd, buf, count)Send data from a buffer to a file/device/pipe/socket. Returns bytes written or −1.
close()int close(fd)Release the descriptor and free kernel resources. Returns 0 on success, −1 on error.

Common open() Flags

O_RDONLY read only O_WRONLY write only O_RDWR read & write O_APPEND write at end O_CREAT create if absent O_EXCL ensure file is new O_TRUNC truncate existing

File permissions (mode) use octal values for Owner / Group / Others — e.g. 0644 → rw- r-- r--. Each open file has a current position called the offset: reading read(fd,buffer,5) on "Hello World" returns "Hello", and the next read continues from where it left off.

Exam Tip — Return Values Remember the special return values: read() returns 0 at EOF; open() returns −1 on failure (a valid fd is otherwise ≥ 0). write(1, "Hello\n", 6) prints to stdout.

Inter-Process Communication (IPC)

Independent vs cooperating processes, why processes cooperate, and the two fundamental IPC models — shared memory and message passing (pipes & sockets).

Independent vs Cooperating Processes

Independent

  • Cannot affect or be affected by other processes.
  • Does not share data with any other process.
vs

Cooperating

  • Can affect or be affected by other processes.
  • Shares data and coordinates with others.

Why Processes Cooperate

IS

Information Sharing

Several users may want the same piece of information (e.g. a shared file).

CS

Computation Speedup

Break a task into subtasks running in parallel — only achievable with multiple cores.

M

Modularity

Divide system functions into separate processes or threads.

Cv

Convenience

An individual user works on many tasks at once — editing, music, compiling.

Two Fundamental IPC Models

Shared Memory

  • A region of memory is shared by cooperating processes.
  • Processes exchange information by reading and writing to the shared region.
  • The region resides in the address space of the creating process; others must attach it to theirs.
vs

Message Passing

  • Communication by explicitly sending and receiving messages through the OS kernel.
  • No direct access to a shared region.
  • Implemented via pipes and sockets.

Shared-Memory POSIX System Calls

CallRole
shm_open()Create / open the shared-memory segment (Linux creates /dev/shm/…).
ftruncate()Set the maximum size of the segment.
mmap()Map process memory to the shared segment via a pointer — used to read & write.
munmap()Unmap process memory after IPC.
shm_unlink()Remove the shared memory.

Pros of Shared Memory

  • Fastest way of IPC.
  • No kernel copying after mmap.
  • Efficient for large data.

Cons of Shared Memory

  • No synchronization — processes must coordinate access themselves.

IPC Using Pipes

Pipe

Pipes provide one-way (half-duplex) communication between related processes, implemented on kernel I/O buffers. A pipe connects the standard output of one process to the standard input of another — the basis of UNIX shell pipelines.

  • Accessed like ordinary files — they have inodes and use file descriptors.
  • The pipe inode resides in the kernel, not the physical file system.
  • A pipe exists only while its creator process exists and has limited size (since Linux 2.6.11, capacity is 65,536 bytes).
Creating & Using Pipes int pipe(int fd[2]) opens two descriptors — fd[0] for reading, fd[1] for writing — returning 0 on success, −1 otherwise. A child inherits open descriptors from the parent, so a pipe lets a process communicate with its child. Both must close unused descriptors, then use read() and write().

IPC Using Sockets

Sockets as the interface between application and network, the eight key socket system calls, the server/client workflow, and socket domains and types.

What is a socket?

A socket is an interface between an application and the network. The application creates a socket, then passes data to it for transmission and receives data from it. Sockets let processes communicate on the same machine or across different machines.

The Server / Client Workflow

The socket calls must be used in a specific order. One process acts as the server, the other as the client; a server may communicate with many clients.

Server Process

  • socket() — create endpoint
  • bind() — attach to IP & port
  • listen() — mark as passive
  • accept() — accept a client
  • send() / recv() — exchange data
  • close() — release
⇄

Client Process

  • socket() — create endpoint
  • connect() — connect to server
  • send() / recv() — exchange data
  • close() — release

The Socket System Calls

CallRole
socket()Creates a process endpoint — socket(domain, type, protocol). Returns a socket fd (>0) or −1.
bind()Associates the socket with a local IP address and port.
listen()Marks the socket passive to accept connections — listen(sockfd, backlog); backlog = max pending requests.
accept()Accepts a client request and creates a new socket for that client; the original keeps listening.
connect()Used by the client to establish a connection with the server.
send() / recv()Send and receive data. recv() returns bytes received, 0 if connection closed, −1 on error.
close()Closes the socket descriptor.

Socket Domains & Types

AF

Domains

AF_INET — IPv4 · AF_INET6 — IPv6 · AF_UNIX — local IPC using file-system pathnames as addresses.

TCP

SOCK_STREAM

TCP sockets — reliable, in-order, connection-oriented delivery.

UDP

SOCK_DGRAM

UDP sockets — unreliable, no order guarantee, no notion of connection.

SOCK_RAW & Standard Ports SOCK_RAW gives direct access to underlying network protocols (bypassing TCP/UDP) — used for new protocols and tools like ping, traceroute, nmap and packet sniffing. Standard ports: HTTP 80, FTP 20/21, POP3 110, SMTP 25, NNTP 119, Telnet 23, IRC 194. Conversion functions htonl/htons/ntohl translate between host and network byte order.

Process Synchronization

Race conditions, the critical-section problem, the producer–consumer problem, the three requirements for a correct solution, and Peterson's software solution.

Race Conditions

When multiple processes/threads run concurrently and share memory, files or variables, accessing shared data without coordination creates a problem.

Race Condition

A situation where the final result depends on the timing/order of threads, not on the logic. For example, two concurrent counter++ operations each compile to load → add → store; if these interleave arbitrarily, the result can be 6 (wrong) instead of the expected 7.

The Critical Section

Critical Section For n processes {P₀ … Pₙ₋₁}, each has a critical section — a code segment that changes shared variables, tables or files. The rule: when one process is in its critical section, no other may be in its own. No two processes execute their critical sections simultaneously.

Each process structures its code into four sections:

Structure of a Process
1
Entry SectionRequests permission to enter the critical section.
2
Critical SectionAccess to shared resources happens here.
3
Exit SectionReleases the critical section.
4
Remainder SectionThe rest of the code.

The Producer–Consumer Problem

Classic synchronization problem

One or more producers generate data, one or more consumers use it, and both share a common buffer. When they run concurrently, two issues arise.

Producer too fast

The buffer becomes full and new data may overwrite old data.

Consumer too fast

The buffer becomes empty and the consumer reads invalid/garbage data.

Three Requirements of a Correct Solution

ME

Mutual Exclusion

If Pᵢ is in its critical section, no other process can be in theirs.

P

Progress

If no one is in the CS and some want in, only contending processes decide who enters next — and the choice cannot be postponed indefinitely.

BW

Bounded Waiting

There is a limit on how many times others enter after a process requests entry and before it is granted.

Peterson's Solution

Software-based

A software algorithm that ensures mutual exclusion between two processes (P₀ and P₁) without hardware support. It works only for two processes.

Why Peterson's solution can fail on modern systems Modern CPUs perform instruction reordering, compiler optimizations, and have cache inconsistency across cores. This reordering + cache delay can break Peterson's solution — leading to race conditions and incorrect synchronization. The fix is hardware support (next section).

Hardware Support, Locks, Semaphores & Monitors

Atomic hardware instructions, mutex locks and spinlocks, condition variables, semaphores (wait/signal), the deadlock danger of lock ordering, and monitors.

Hardware Support for Synchronization

Hardware support provides atomic and ordered operations to safely manage shared data. Three types:

MB

Memory Barriers

Ensure memory operations are executed in order.

HI

Hardware Instructions

Special atomic CPU instructions — Test-and-Set and Compare-and-Swap (CAS).

AV

Atomic Variables

Variables supporting indivisible operations completing in one step, with no interruption.

The Concept of a Lock

Lock

A lock is a variable used to ensure that only one process/thread enters the critical section at a time.

Why a naive lock fails The two steps if (lock == 0) then lock = 1; are NOT atomic — two threads can both pass the check. We need an operation that does CHECK + SET in one atomic step: that's Test-and-Set. Compare-and-Swap compares a value and updates it only if it matches the expected value.

Mutex Locks

Mutex (Mutual Exclusion Lock)

A synchronization mechanism ensuring only one thread enters the critical section at a time. Protect a CS by first acquire()-ing the lock, then release()-ing it — both must be atomic (via hardware instructions like CAS).

Spinlock (Busy Waiting)

  • The thread keeps checking — busy-waiting — so it's called a spinlock.
  • Used for kernel internals and short critical sections.
vs

Blocking Mutex (Sleeping)

  • The thread goes to sleep and the OS wakes it later.
  • Modern mutexes (pthread) try fast → spin briefly → sleep (futex).

The pthread interface: pthread_mutex_init, pthread_mutex_lock (blocks if locked), pthread_mutex_trylock (returns "busy" immediately), pthread_mutex_unlock and pthread_mutex_destroy. Mutex variables are of type pthread_mutex_t and must be initialized before use.

Exam Tip — Lock Ordering & Deadlock If Thread 1 needs lock x then y, while Thread 2 needs lock y then x, they can deadlock. The fix is a consistent lock ordering — every thread acquires locks in the same global order.

Condition Variables

Mutex ensures mutual exclusion but does not tell a thread when to proceed — it provides no condition-based synchronization.

Condition Variable

Allows a thread to sleep until a specific condition becomes true. The producer–consumer solution uses two condition variables — one for "buffer not full", one for "buffer not empty".

Semaphores

Semaphore

A more sophisticated synchronization tool than mutex locks. A semaphore S is an integer accessed only via two indivisible (atomic) operations: wait() and signal() — originally called P() and V().

wait(S) and signal(S)

wait(S): while (S ≤ 0) busy-wait; then S−−.
signal(S): S++.

Counting vs Binary

  • Counting semaphore — value over an unrestricted domain (e.g. 3 printers available).
  • Binary semaphore — value 0 or 1, the same as a mutex lock.

The C library: sem_init(&sem, 0, value) (0 = shared between threads, value = initial count), sem_wait(&sem) and sem_post(&sem).

Common semaphore mistakes signal(mutex) … wait(mutex) (reversed), wait(mutex) … wait(mutex) (double-wait → deadlock), or omitting wait or signal entirely. These show what goes wrong when synchronization tools are used incorrectly.

Semaphore vs Mutex

Use a Mutex

  • Protecting shared data.
  • Ownership matters — the same thread that locks must unlock.
vs

Use a Semaphore

  • Signaling between threads.
  • Resource counting.
  • A different thread can wake another.

Monitors

Monitor

A high-level abstraction providing a convenient mechanism for process synchronization. It is an abstract data type whose internal variables are only accessible by code within its procedures, and only one process may be active inside the monitor at a time.

Threads & Multithreading

Why threads, their benefits, concurrency vs parallelism, Amdahl's Law, user vs kernel threads, the three multithreading models, thread libraries, and threading issues.

Motivation & Benefits

Most modern applications are multithreaded — separate threads handle separate tasks (update display, fetch data, spell check, answer a network request). Process creation is heavy-weight; thread creation is light-weight.

R

Responsiveness

Execution may continue even if part of the process is blocked — vital for user interfaces.

RS

Resource Sharing

Threads share the process's resources — easier than shared memory or message passing.

E

Economy

Cheaper than process creation; thread switching has lower overhead than context switching.

S

Scalability

A process can take advantage of multicore architectures.

Concurrency vs Parallelism

Concurrency

  • Supports more than one task making progress.
  • Achievable on a single core via a scheduler.
vs

Parallelism

  • The system performs more than one task simultaneously.
  • Requires multiple cores.

Two types of parallelism: data parallelism distributes subsets of the same data across cores (same operation on each), while task parallelism distributes threads across cores, each performing a unique operation.

Amdahl's Law

Amdahl's Law Identifies performance gains from adding cores when an application has both serial (S) and parallel parts. If an app is 75% parallel / 25% serial, moving from 1→2 cores gives ~1.6× speedup. As cores N → ∞, speedup approaches 1 / S. The serial portion has a disproportionate effect on the gains.

User Threads vs Kernel Threads

User Threads

Managed by a user-level threads library. Three primary libraries: POSIX Pthreads, Windows threads, Java threads.

Kernel Threads

Supported directly by the kernel. Found in virtually all general-purpose OSes — Windows, Linux, macOS, iOS, Android.

The Three Multithreading Models

M:1

Many-to-One

Many user threads → one kernel thread. One blocking call blocks all; no true parallelism. Rare today (e.g. GNU Portable Threads).

1:1

One-to-One

Each user thread maps to a kernel thread — more concurrency, but overhead limits the count. Used by Windows & Linux.

M:M

Many-to-Many

Many user threads mapped to many kernel threads — the OS creates a sufficient number. The two-level model also lets a user thread bind to a kernel thread.

Pthreads & Implicit Threading

Pthreads

A POSIX standard (IEEE 1003.1c) API for thread creation and synchronization. It is a specification, not an implementation — common in UNIX, Linux and macOS. Java threads are managed by the JVM, typically using the underlying OS model; standard practice is to implement the Runnable interface.

Implicit threading moves thread creation and management to compilers and run-time libraries. Five methods: Thread Pools, Fork-Join, OpenMP, Grand Central Dispatch, and Intel Threading Building Blocks (TBB).

Thread Pools Create a number of threads in a pool that await work. Benefits: servicing a request with an existing thread is faster than creating one; the number of threads is bounded by the pool size; and tasks are decoupled from thread creation.

Threading Issues

⑂

fork() & exec()

Does fork() duplicate one thread or all? exec() usually replaces all threads.

⚑

Signal Handling

To which thread is a signal delivered? To the applicable thread, every thread, certain threads, or one assigned thread.

✕

Thread Cancellation

Terminating a target thread — asynchronous (immediate) or deferred (checked at cancellation points). Default is deferred.

TLS

Thread-Local Storage

Each thread gets its own copy of data — visible across function invocations, unique per thread.

SA

Scheduler Activations

Provide upcalls from kernel to the thread library to maintain the right number of kernel threads, via lightweight processes (LWPs).

L

Linux Tasks

Linux calls them tasks; creation uses clone(), whose flags control how much the child shares with the parent.

CPU Scheduling Algorithms

The key scheduling metrics, and the major scheduling algorithms — FCFS, SJF, Priority and Round Robin — with the formulas for completion, turnaround and waiting time.

Scheduling Metrics

CT

Completion Time

The time at which a process finishes execution.

TAT

Turnaround Time

TAT = Completion Time − Arrival Time. Total time from arrival to completion.

WT

Waiting Time

WT = Turnaround Time − Burst Time. Time spent waiting in the ready queue.

Scheduling Algorithms

AlgorithmTypeHow it works
FCFS — First Come First ServeNon-preemptiveProcesses run in arrival order. Simple but suffers the convoy effect — short jobs wait behind long ones.
SJF — Shortest Job FirstNon-preemptiveRuns the process with the shortest burst time next. Optimal average waiting time but needs to know burst lengths.
SRTF — Shortest Remaining Time FirstPreemptiveThe preemptive version of SJF — a newly arrived shorter job preempts the running one.
Priority SchedulingBothThe highest-priority process runs first. Risk of starvation, solved by aging.
Round Robin (RR)PreemptiveEach process gets a fixed time quantum; ideal for time-sharing. Fair, with bounded waiting.
Exam Tip — Solving Scheduling Problems For any numerical: draw the Gantt chart first, then read off Completion Times. Compute TAT = CT − AT and WT = TAT − BT for each process, then take the averages. FCFS is non-preemptive (arrival order); RR uses the time quantum.
11 Unit II · Memory

Memory Management

Background and protection, address binding, logical vs physical addresses, the MMU, dynamic loading & linking, and the contiguous allocation schemes — fixed and dynamic partitioning with their fit algorithms.

Background

  • A program must be brought from disk into memory and placed within a process to run.
  • Register access takes one CPU clock; main memory can take many cycles; a cache sits between them.
  • Protection of memory is required for correct operation.
Base & Limit Registers

A process must access only addresses in its own address space. A pair of base and limit registers defines its logical address space; the CPU checks every user-mode access against them. The instructions that load these registers are privileged.

Address Binding

Addresses are represented differently at different stages of a program's life — source code uses symbolic addresses, compiled code uses relocatable addresses, and the linker/loader binds them to absolute addresses. Binding can happen at three stages:

①

Compile Time

If the location is known in advance, absolute code is generated — but the code must be recompiled if the start location changes.

②

Load Time

If the location is unknown at compile time, relocatable code is generated.

③

Execution Time

Binding is delayed until run time so the process can be moved during execution — needs hardware support (base/limit registers).

Logical vs Physical Address Space

Logical (Virtual) Address

  • Generated by the CPU.
  • The set of all such addresses is the logical address space.
→

Physical Address

  • The address seen by the memory unit.
  • They are the same under compile/load-time binding, but differ under execution-time binding.
Memory-Management Unit (MMU) A hardware device that maps virtual → physical addresses at run time. In the simple relocation-register scheme, the register's value is added to every address a user process generates. The user program never sees real physical addresses.

Dynamic Loading & Dynamic Linking

Dynamic Loading

A routine is not loaded until it is called — better memory utilization, since unused routines are never loaded. Useful for large, infrequently-used code. No special OS support required.

Dynamic Linking

Linking is postponed until execution time. A small stub locates the memory-resident library routine and replaces itself with the routine's address. Also known as shared libraries.

Memory Management Techniques

Two Families of Allocation

Contiguous

The executing process must be loaded entirely in main memory.

  • Fixed (static) partitioning
  • Variable (dynamic) partitioning

Non-Contiguous

The process is loaded into non-contiguous locations.

  • Paging
  • Segmentation

Fixed (Static) Partitioning

  • The number of partitions is fixed; sizes may or may not be equal.
  • Being contiguous, no spanning is allowed.
  • The number of partitions defines the degree of multiprogramming.
  • Suffers from internal fragmentation.

Partition Allocation (Fit) Algorithms

1st

First Fit

Allocate the first sufficient block from the top of memory.

B

Best Fit

Allocate the smallest sufficient free partition.

W

Worst Fit

Allocate the largest sufficient free partition.

N

Next Fit

Like first fit, but searches from the last allocation point.

Dynamic (Variable) Partitioning

Partitions are created at run-time according to each process's need — the partition size equals the incoming process, so the number of partitions is not fixed.

Advantages

  • No internal fragmentation — space allocated exactly to need.
  • No restriction on the degree of multiprogramming.
  • No limit on process size (vs largest fixed partition).

Disadvantages

  • Difficult to implement — allocation at run-time.
  • External fragmentation, requiring a compaction algorithm.
Solving the storage-allocation problem Three remedies for fragmentation: a Hole is a block of available memory; Coalescing merges adjacent free holes; Compaction shuffles memory to place all free space together.
12 Unit II · Memory

Paging & Segmentation

Non-contiguous allocation — paging with its address-translation scheme, the page table and TLB, and segmentation with its segment table, plus a side-by-side comparison.

Paging

Paging

Allows a process's logical address space to be non-contiguous. Physical memory is divided into fixed-size frames (a power of 2, between 512 B and 8 KB); logical memory into same-size pages. To run a program of n pages, find n free frames and set up a page table to translate logical to physical addresses. Internal fragmentation is possible.

Address-Translation Scheme

Page Number + Page Offset The CPU-generated address splits into a page number (p) — an index into the page table holding each page's base address — and a page offset (d), combined with the base to give the physical address. For a logical space of 2ᵐ and page size 2ⁿ, p uses (m−n) bits and d uses n bits.

Implementing the Page Table

  • The page table is kept in main memory; PTBR points to it and PTLR indicates its length.
  • Naively, every access needs two memory accesses — one for the table, one for the data.
  • This is solved by the TLB (Translation Look-aside Buffer) — a fast associative-memory cache.
TLB

On a TLB miss, the page table is consulted and the <page, frame> pair is added to the TLB (replacing an entry if full). Some entries (kernel code) are wired down. ASIDs (address-space identifiers) uniquely identify each process so the TLB need not be flushed on every context switch.

Segmentation

Segmentation

A memory-management scheme that supports the user's view of memory. A program is a collection of segments — logical units such as the main program, procedures, functions, objects, local & global variables, stack, symbol table and arrays.

A logical address is a tuple <segment-number, offset>. The segment table maps it: each entry has a base (starting physical address) and a limit (segment length). STBR points to the table; STLR gives the number of segments — segment s is legal if s < STLR.

Advantages of Segmentation

  • No internal fragmentation.
  • Segment tables consume less memory than page tables (one entry per segment).
  • Lends itself to sharing data among processes.
  • Each segment can be set its own access right.

Disadvantages

  • External fragmentation.
  • Costly memory-management algorithm.
  • Unequal segment sizes are awkward for swapping.
Exam Tip — Paging vs Segmentation Paging = fixed-size pages, invisible to the user, suffers internal fragmentation. Segmentation = variable-size logical units matching the user's view, suffers external fragmentation. Both are non-contiguous.
13 Unit II · Memory

Virtual Memory & Demand Paging

The idea of virtual memory, demand paging with the lazy swapper, the valid–invalid bit, and the step-by-step handling of a page fault.

What is Virtual Memory?

Virtual Memory

The separation of user logical memory from physical memory. Only part of a program needs to be in memory to execute, so the logical address space can be much larger than physical memory.

  • Only part of the program needs to be in memory — less memory needed.
  • Faster response and support for more users.
  • Allows address spaces to be shared by several processes.
  • Allows for more efficient process creation.

Virtual memory can be implemented via demand paging or demand segmentation.

Demand Paging

The Lazy Swapper Demand paging brings a page into memory only when it is needed. A lazy swapper never swaps a page in unless it will be needed — a swapper that deals with pages is called a pager.

The Valid–Invalid Bit

Valid–Invalid Bit

Each page-table entry has a valid–invalid bit: v = in-memory, i = not-in-memory. Initially all entries are set to i. During address translation, if the bit is i, a page fault traps to the OS.

Steps in Handling a Page Fault

Page-Fault Service Routine
1
Check the TableThe OS checks another table — an invalid reference aborts; a valid one is just not yet in memory.
2
Get an Empty FrameFind a free frame in physical memory.
3
Swap the Page InBring the desired page into the frame from disk.
4
Reset the TablesUpdate the page and frame tables; set the validation bit to v.
5
Restart the InstructionRe-run the instruction that caused the fault.
No free frame? If there is no free frame, the OS uses a page-replacement algorithm to find a page in memory that is not really in use and swaps it out. The same page may be brought into memory several times.
14 Unit II · Memory

Page Replacement & Thrashing

Basic page replacement, the three classic algorithms — FIFO, Optimal and LRU — with their page-fault behaviour, plus thrashing.

Basic Page Replacement

Replacing a Victim Frame
1
Locate the PageFind the desired page on disk.
2
Find a FrameIf free, use it; otherwise use a replacement algorithm to select a victim frame.
3
Bring In the PageLoad the desired page into the newly free frame; update the page and frame tables.
4
Restart the ProcessResume execution.

We want the lowest page-fault rate. Algorithms are evaluated by running them on a reference string of memory references and counting page faults. The standard example string is 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5.

The Three Classic Algorithms

FIFO

First-In-First-Out

Replaces the oldest page. Simple but can perform poorly — and exhibits Belady's anomaly (more frames can mean more faults). On the example: 9 faults with 3 frames, 10 faults with 4.

OPT

Optimal

Replaces the page not used for the longest future period. Gives the fewest faults (6 on the example) but requires future knowledge — used mainly as a benchmark.

LRU

Least Recently Used

Replaces the page not used for the longest past period — an approximation of Optimal. Implemented with a counter per page entry that records the clock at each reference.

Exam Tip — Page-Fault Counts Memorise the example string 1,2,3,4,1,2,5,1,2,3,4,5: with the same frames, OPT ≤ LRU ≤ FIFO in faults. OPT looks forward; LRU looks backward; FIFO ignores usage. Watch for Belady's anomaly in FIFO.

Thrashing

Thrashing

When a process does not have enough frames, it spends more time paging than executing — page faults follow one another so closely that the CPU utilization collapses. The OS may increase the degree of multiprogramming and make it worse; the working-set model helps allocate enough frames to avoid it.

15 Unit III · Deadlocks

Deadlocks

The four necessary conditions, the four ways of handling deadlocks — prevention, avoidance (Banker's Algorithm), detection and recovery.

What is a Deadlock?

Deadlock

A set of processes is deadlocked when every process in the set is waiting for an event that only another process in the set can cause — typically the release of a resource. None can proceed.

The Four Necessary Conditions

A deadlock can arise only if all four hold simultaneously:

①

Mutual Exclusion

At least one resource is held in a non-sharable mode.

②

Hold and Wait

A process holding resources is waiting to acquire more held by others.

③

No Preemption

Resources cannot be forcibly taken — only released voluntarily.

④

Circular Wait

A circular chain of processes exists, each waiting for a resource held by the next.

Four Ways to Handle Deadlocks

StrategyIdea
PreventionEnsure at least one of the four necessary conditions cannot hold — e.g. eliminate hold-and-wait or impose a total ordering to break circular wait.
AvoidanceUse advance information about resource needs to keep the system in a safe state — the Banker's Algorithm.
DetectionAllow deadlocks to occur, then run a detection algorithm (a wait-for graph) to find them.
RecoveryBreak a detected deadlock by process termination or resource preemption (rolling back a victim).

Deadlock Avoidance — Banker's Algorithm

Safe State The Banker's Algorithm grants a resource request only if doing so leaves the system in a safe state — one where there exists a safe sequence in which every process can finish. A safe state is never a deadlocked state, though not every unsafe state is a deadlock.
Exam Tip — Remember "M-H-N-C" The four conditions: Mutual Exclusion, Hold-and-Wait, No Preemption, Circular Wait. Prevention attacks one condition; avoidance keeps a safe state; detection + recovery lets it happen then fixes it.
16 Unit III · I/O & Storage

Disk & Storage Management

Disk geometry, the components of disk access time, and the five disk-scheduling algorithms — FCFS, SSTF, SCAN, C-SCAN and LOOK — with a worked access-time example.

Disk Structure

Magnetic disks provide the bulk of secondary storage. Data is stored on a stack of rotating magnetic platters in concentric tracks; read/write heads move radially to position over the desired track.

S

Sector

The smallest unit that can be read or written — data is stored in equal-size blocks (512 to 4096 bytes).

T

Track

A circle of consecutive sectors on one surface, along one revolution.

C

Cylinder

A collection of tracks at the same radius across multiple surfaces — accessing within a cylinder needs only a faster head switch, not head movement.

Disk Access Time

Access Time = Seek + Rotational Latency + Transfer Seek time — time to move the head to the desired track. Rotational latency — time for the desired sector to rotate under the head (average = ½ rotation). Transfer time = transfer size ÷ transfer rate.
Worked Example

Given transfer size 8 KB, average seek 12 ms, 7200 RPM, transfer rate 4 MB/s:
Rotational latency = (60 / 7200) ÷ 2 = 8.33 ÷ 2 = 4.16 ms.
Transfer time = 8 / 4000 = 0.002 s = 2 ms.
Total access time = 12 + 4.16 + 2 = 18.16 ms.

Disk Scheduling

The OS schedules disk I/O requests in a good order to improve access time and bandwidth (total bytes transferred ÷ total time). Using the standard queue 98, 183, 37, 122, 14, 124, 65, 67 with the head at cylinder 53:

AlgorithmHow it works
FCFSServices requests in arrival order. Fair, but slow — the example gives 640 cylinders of head movement.
SSTF (Shortest Seek Time First)Services the request closest to the current head. Better than FCFS, but can cause starvation.
SCAN (Elevator)The arm moves to one end servicing requests, then reverses — scanning back and forth like an elevator.
C-SCANLike SCAN, but on reaching the end it jumps back to the start without servicing on the return — giving more uniform wait time.
LOOKLike SCAN but the arm only goes as far as the last request in each direction, not the physical end of the disk.
Exam Tip — Choosing an Algorithm SSTF is common and naturally appealing (better than FCFS). SCAN and C-SCAN perform better under heavy load because they avoid starvation. The scheduler should be a separate module so it can be swapped out.
17 Unit IV · Files

File Management

What a file is, file attributes and operations, file types and access methods, directory structures, and the OS's file-system management activities.

The File Concept

File

The OS provides a uniform, logical view of information storage, abstracting the physical properties of storage devices into a logical unit — the file. Each medium (disk drive, tape drive) is controlled by a device with varying access speed, capacity, transfer rate and access method.

File Attributes & Operations

Attributes

Name, identifier, type, location, size, protection, and time/date/owner information — stored in the directory structure.

Operations

Create, write, read, reposition (seek), delete and truncate — the six basic operations the OS provides via system calls.

File Access Methods

SEQ

Sequential

Information is processed in order, one record after another — the most common method.

DIR

Direct (Relative)

The file is a numbered sequence of blocks/records read or written in any order — fast random access.

IDX

Indexed

An index is built containing pointers to blocks; the index is searched, then the block is accessed directly.

Directory Structures

StructureDescription
Single-LevelAll files in one directory — simple, but naming conflicts and no grouping.
Two-LevelA separate directory per user — solves naming conflicts between users.
Tree-StructuredA hierarchy of directories and subdirectories — the most common, allowing grouping.
Acyclic-GraphAllows shared subdirectories and files (via links) — but no cycles.
General GraphPermits cycles — needs garbage collection to reclaim space.
File-System Management Activities Files are usually organized into directories with access control determining who can access what. The OS handles: creating & deleting files and directories, primitives to manipulate them, mapping files onto secondary storage, and backing up files onto stable storage.
18 Unit IV · Security

OS Security, Virtualization & Sandboxing

Protection vs security, access control, the role and types of hypervisors, container virtualization with Docker, sandboxing, and the computing environments.

Protection vs Security

Protection

  • Any mechanism for controlling access of processes or users to resources defined by the OS.
  • Internal — about who can do what.
vs

Security

  • Defense against internal and external attacks.
  • Covers denial-of-service, worms, viruses, identity & service theft.

Access Control

Systems distinguish among users to determine who can do what:

  • User IDs (security IDs) — a name and number, one per user, associated with all that user's files and processes for access control.
  • Group IDs — define a set of users so controls can be managed collectively.
  • Privilege escalation — allows a user to change to an effective ID with more rights.

Virtualization

Virtualization

Allows operating systems to run applications within other OSes. It typically refers to creating a virtual machine that virtualizes all hardware resources — processors, memory, storage and network — so physical hardware can be shared by one or more VMs.

Benefits: consolidation to reduce hardware cost, optimization of workloads, and IT flexibility and responsiveness. Emulation is used when the source CPU type differs from the target (e.g. PowerPC → x86) and is the slowest method.

The Hypervisor (VMM)

Hypervisor / Virtual Machine Monitor A software layer that lets multiple Guest OSes (VMs) run simultaneously on a single physical host, providing hardware abstraction and efficiently multiplexing the underlying resources. A VMM manages CPU, memory, I/O, interrupts and the instruction set; a hypervisor may refer to an OS with the VMM.

Type 1 Hypervisor

  • Runs directly on the hardware (bare metal).
  • Examples: Xen Server.
vs

Type 2 Hypervisor

  • Runs on top of a host OS.
  • Examples: VirtualBox on Windows.

Container Virtualization & Docker

Containers (OS-Level Virtualization)

Containers use the host operating system as their base, not a hypervisor. They virtualize the OS itself, sharing the host OS kernel and its resources. Isolation is provided by kernel namespaces — global resources are wrapped so they're visible only to processes in the same namespace.

Docker is a collection of open-source tools, solutions and cloud services that provide a common model for packaging (containerizing) application code into images that can be easily distributed to and deployed on Linux hosts.

Sandboxing

Sandboxing A sandbox is a tightly controlled, isolated environment in which untrusted code can run with restricted access to the rest of the system — limiting the damage malicious or buggy programs can do. Namespaces and containers are practical applications of the same isolation principle.

Computing Environments

T

Traditional

Stand-alone general-purpose machines, now mostly interconnected via the Internet behind firewalls.

M

Mobile

Handhelds with extra features (GPS, gyroscope), using 802.11 or cellular. Leaders: iOS & Android.

CS

Client–Server

Servers respond to requests from clients — compute-server and file-server systems.

P2P

Peer-to-Peer

No client/server distinction — all nodes are peers (Napster, Gnutella, Skype/VoIP).

☁

Cloud Computing

Delivers computing, storage and apps as a service — built on virtualization. Types: public, private, hybrid; SaaS, PaaS, IaaS.

RT

Real-Time Embedded

The most prevalent form of computers, with well-defined fixed time constraints that must be met for correct operation.

Exam Tip — VM vs Container A hypervisor runs full guest OSes (each with its own kernel) — heavier but stronger isolation. A container shares the host kernel via namespaces — lighter and faster but with weaker isolation. Both rest on the same goal: sharing hardware safely.
