Introduction to Operating Systems

What an operating system is, what goals it serves, the four components of a computer system, and the many roles the term "OS" covers — from toasters to spacecraft.

What is an Operating System?

An operating system is a program that acts as an intermediary between a user of a computer and the computer hardware.

The Three Goals of an OS Execute user programs and make solving user problems easier · Make the computer convenient to use · Use the hardware efficiently.

Computer System Structure — Four Components

A computer system can be divided into four components, each layered on the one below it:

①

Hardware

Provides the basic computing resources — CPU, memory, and I/O devices.

②

Operating System

Controls and coordinates the use of hardware among various applications and users.

③

Application Programs

Define the ways system resources solve users' problems — word processors, compilers, browsers, databases, games.

④

Users

People, machines, and other computers that use the system.

What Operating Systems Do — A Matter of Viewpoint

What an OS does depends on the point of view:

  • Users want convenience, ease of use and good performance — they don't care about resource utilization.
  • Shared computers (mainframe/minicomputer) must keep all users happy — here the OS is a resource allocator and control program.
  • Dedicated systems (workstations) have dedicated resources but use shared resources from servers.
  • Mobile devices are resource-poor — optimized for usability and battery life, using touch screens and voice.
  • Embedded computers (devices, automobiles) have little or no user interface and run without intervention.

Defining the Operating System

No universally accepted definition

A good approximation: "everything a vendor ships when you order an operating system" — but this varies wildly.

K

Kernel

The one program running at all times on the computer — the core of the OS.

SP

System Program

Ships with the OS, but is not part of the kernel.

AP

Application Program

All programs not associated with the operating system.

Modern general-purpose and mobile OSes also include middleware — a set of software frameworks providing additional services (databases, multimedia, graphics) to application developers.

Where OSes Live Operating systems are present in toasters through ships, spacecraft, game machines, TVs and industrial control systems. The OS was born when fixed-use military computers became general-purpose and needed resource management and program control.

Computer-System Structure & Operations

How the hardware is organised, the central role of interrupts, the storage hierarchy, multiprocessor architectures, bootstrapping, multiprogramming vs multitasking, and dual-mode operation.

Computer-System Organization

One or more CPUs and device controllers connect through a common bus providing access to shared memory, executing concurrently and competing for memory cycles.

  • Each device controller is in charge of a particular device type and has a local buffer.
  • Each controller type has an OS device driver to manage it.
  • The CPU moves data between main memory and local buffers; I/O is from the device to the controller's local buffer.
  • The controller informs the CPU it has finished by causing an interrupt.

Interrupts — The Heart of the OS

An operating system is interrupt-driven An interrupt transfers control to the interrupt service routine via the interrupt vector (a table of service-routine addresses). The architecture must save the address of the interrupted instruction.

Interrupt Handling

  • The OS preserves the CPU state by storing the registers and program counter.
  • It determines which type of interrupt occurred.
  • Separate segments of code decide the action for each type.

Traps & Exceptions

A trap (or exception) is a software-generated interrupt caused either by an error (e.g. division by zero) or a user request (a system call).

I/O Structure & DMA

There are two methods for handling I/O after it starts:

Synchronous (blocking)

  • Control returns to the user program only upon I/O completion.
  • A wait instruction idles the CPU until the next interrupt.
  • At most one I/O request outstanding at a time — no simultaneous processing.
vs

Asynchronous

  • Control returns without waiting for I/O completion.
  • A system call lets the user wait for completion if desired.
  • A device-status table holds each device's type, address and state.
Direct Memory Access (DMA)

Used for high-speed I/O devices able to transmit near memory speed. The device controller transfers blocks of data directly from its buffer to main memory without CPU intervention — generating only one interrupt per block instead of one per byte.

Storage Hierarchy

Storage systems are organised in a hierarchy by speed, cost and volatility. Caching — copying information into faster storage — appears at every level; main memory can be viewed as a cache for secondary storage.

▲

Main Memory

The only large storage the CPU can access directly. Random-access, typically volatile DRAM.

HDD

Hard Disk Drives

Rigid platters with magnetic recording material; surface divided into tracks subdivided into sectors.

NVM

Non-Volatile Memory

Faster than hard disks and nonvolatile — increasingly popular as capacity rises and price drops.

Storage units The basic unit is the bit (0 or 1). A byte = 8 bits. A word is the architecture's native data unit (e.g. 8 bytes on a 64-bit machine). KB = 1,024 bytes; MB = 1,024² ; GB = 1,024³ ; TB = 1,024⁴ ; PB = 1,024⁵. Networking is the exception — measured in bits.

Computer-System Architecture

Multiprocessor (Parallel) Systems

Also called tightly-coupled systems. Advantages:

  • Increased throughput
  • Economy of scale
  • Increased reliability — graceful degradation / fault tolerance

Clustered Systems

Multiple systems working together, usually sharing storage via a SAN, providing a high-availability service:

  • Asymmetric — one machine in hot-standby mode.
  • Symmetric — multiple nodes running and monitoring each other.
  • Some are for HPC; some use a distributed lock manager (DLM).

Multiprogramming vs Multitasking

Multiprogramming (Batch)

  • A single user cannot always keep the CPU and I/O devices busy.
  • Organizes jobs so the CPU always has one to execute.
  • A subset of jobs is kept in memory; one is run via job scheduling.
  • When a job must wait (e.g. for I/O), the OS switches to another job.
→

Multitasking (Timesharing)

  • The CPU switches jobs so frequently that users can interact with each — response time < 1 second.
  • Several ready jobs ⟶ CPU scheduling.
  • If processes don't fit in memory, swapping moves them in and out.
  • Virtual memory allows execution of processes not completely in memory.

Dual-Mode Operation

Exam Tip — User Mode vs Kernel Mode A hardware mode bit distinguishes user mode from kernel mode, letting the OS protect itself. A system call changes the mode to kernel; the return resets it to user. Privileged instructions are executable only in kernel mode — so users cannot set the mode bit to "kernel" themselves.

Working of a Computer System

Von Neumann architecture A modern computer follows the von Neumann model: program instructions and data share the same main memory, and the CPU repeatedly fetches an instruction, decodes it, and executes it — the fetch–decode–execute (stored-program) cycle.

At the lowest level the machine repeats a simple loop:

2
CPU executes the instructionThe operation is carried out on the CPU.
3
If I/O is required, the device controller handles itThe CPU hands the work to the relevant controller.
4
The device sends an interrupt when the work is completeIt signals the CPU that the operation has finished.
5
The OS handles the interrupt and resumes the processControl returns to the interrupted program.

Physically these parts sit on the PC motherboard, which holds the CPU socket, memory slots, buses and device controllers. In a multiprogrammed system main memory is laid out with the OS in one region and several user processes occupying separate regions alongside it.

Computer System Organization

Computer system organization explains how the CPU, memory and input/output devices are connected and how they work together to execute programs. A modern computer system consists of:

  • One or more CPUs
  • Device controllers
  • Shared memory
  • A system bus

All components are connected through a common system bus, which provides shared access to memory.

Buffer

Definition A buffer is a temporary storage area in memory used to hold data while it is being transferred from one place to another. It handles the difference in speed between two devices or processes.

Need for a Buffer

In a computer system different components operate at different speeds — the CPU is very fast, while I/O devices (disk, keyboard, printer) are slower. If data is transferred directly, the CPU may have to wait and system performance decreases. A buffer solves this.

Device Driver

A device driver is a special program that allows the operating system to communicate with a hardware device. It acts as an interface between the OS and the device controller.

  • Each device type has its own device driver.
  • It translates OS commands into hardware-specific instructions.
  • It controls and manages device operations.
  • It helps in handling interrupts generated by the device.
Examples Printer driver → controls the printer · Graphics driver → controls the monitor · USB driver → controls USB devices.

Middleware

A middleware is a set of software frameworks that provide additional services to application developers, such as databases, multimedia and graphics — sitting between the OS and the applications.

Interrupts (by I/O devices)

Definition An interrupt is a signal sent to the CPU by an I/O device (or software) to indicate that an event has occurred and requires immediate attention. The OS is interrupt-driven.

When an I/O device (such as a keyboard, disk or printer) completes its operation, it sends an interrupt signal to the CPU. This causes:

  1. The CPU to stop executing the current program temporarily.
  2. The CPU to save the current state (program counter and registers).
  3. Control to transfer to the Interrupt Service Routine (ISR).
  4. After handling, the CPU resumes execution of the interrupted program.

Trap / Exception

A trap (or exception) is a software-generated interrupt caused by either an error (e.g. division by zero) or a user request (a system call).

Interrupt-Driven I/O Cycle

The I/O cycle, driver to interrupt handler and back

Caching

Caching is copying information into faster storage. Main memory can be viewed as a cache for secondary storage.

Migration of data & cache coherency Because the same datum can exist at several levels (disk → memory → cache → register), the system must always use the most recent value. A multitasking system must fetch the latest copy regardless of where it sits; a multiprocessor system needs cache coherency in hardware so every CPU sees the newest value; in a distributed system the problem is harder still, as several copies of a datum may exist.

Storage Hierarchy

Storage is organised by speed and cost — fast, small, volatile memory at the top; slow, large, non-volatile storage at the bottom. Going down: storage capacity grows and access time increases.

From registers (fastest) down to magnetic tape (largest)

Computer System Architecture

Modern systems use more than one processor. The main arrangements are symmetric multiprocessing, dual-core (multicore), non-uniform memory access, and clustered systems.

1. Symmetric Multiprocessing (SMP)

Two or more identical processors, each with its own registers and cache, share a single main memory through the bus. All processors are peers — there is no master.

SMP — two peer processors sharing main memory

2. Dual-Core Design

Two CPU cores are placed on a single chip. Each core has its own registers and L1 cache, and the cores share an L2 cache and main memory. Terms: multi-chip and multicore; a chassis may contain multiple separate systems.

Dual-core — two cores on one processor sharing L2 cache

3. Non-Uniform Memory Access (NUMA)

Each CPU has its own local memory, and the CPUs are joined by an interconnect. A CPU reaches its own memory quickly and another CPU's memory more slowly — hence "non-uniform" access times.

NUMA — each CPU has local memory, linked by an interconnect

4. Clustered Systems

Several independent computers are linked by an interconnect and share storage through a storage-area network (SAN), working together as one system for higher availability and performance.

Clustered system — computers sharing a storage-area network
Quick recall SMP = many equal CPUs, one shared memory · Dual-core = many cores on one chip, shared L2 · NUMA = each CPU has its own memory, linked by interconnect (access times differ) · Clustered = separate computers sharing storage over a SAN.

The Kernel & Its Types

The kernel is the heart of every operating system. This topic covers what a kernel does, the functions it controls, and the five major kernel architectures — monolithic, microkernel, hybrid, exo-kernel and nano-kernel — with diagrams comparing how each places OS services relative to user and kernel space.

What is a Kernel?

The kernel is the central core of the operating system — the one program that runs at all times, with complete control over everything in the system.

Definition The kernel is the bridge between application software and the hardware. It loads first at boot, stays resident in main memory, and runs in a privileged kernel mode so it can directly access hardware that user programs cannot.

Functions of the Kernel

①

Process Management

Creation, scheduling, context switching and termination of processes.

②

Memory Management

Allocating and freeing memory, address translation, and virtual memory.

③

Device Management

Controlling hardware through device drivers and handling interrupts.

④

System Calls / IPC

Providing the system-call interface and inter-process communication.

User Space vs Kernel Space

Memory is split into two regions. Understanding this split is the key to understanding every kernel type below — the architectures differ mainly in how many services they place in kernel space versus user space.

User Space ↔ Kernel Space
Exam Tip The fundamental trade-off across kernel types: more code in kernel space = faster (fewer mode switches) but less reliable (one bug can crash the whole OS). More code in user space = more reliable & modular but slower (frequent message passing across the boundary).

The Five Types of Kernel

Kernels are classified by how OS services are organised relative to the user–kernel boundary.

1 Monolithic Kernel 2 Microkernel 3 Hybrid Kernel 4 Exokernel 5 Nanokernel

① Monolithic Kernel  vs  ② Microkernel

Monolithic Kernel All-in-kernel

The entire OS runs in kernel space as a single large program. Scheduling, memory, file system, drivers and IPC all sit in one address space and call each other directly as function calls.

Fast (direct calls) High performance One bug can crash OS Large & hard to maintain

Examples: Linux, traditional UNIX, MS-DOS, older Windows.

Microkernel Minimal kernel

Only the bare essentials run in kernel space (IPC, basic scheduling, low-level memory). Other services — drivers, file system, networking — run as separate user-space servers that talk via message passing.

Reliable & stable Modular, easy to extend Slower (message passing) More context switches

Examples: Mach, MINIX, QNX, L4, Symbian.

③ Hybrid · ④ Exokernel · ⑤ Nanokernel

③

Hybrid Kernel

A compromise between monolithic and microkernel. The structure is microkernel-like, but performance-critical services (e.g. drivers, file system) are pulled back into kernel space to avoid the message-passing penalty.

Examples: Windows NT/10/11, macOS (XNU), modern BSD.

④

Exokernel

Pushes abstraction out of the kernel entirely. The exokernel only securely allocates hardware resources; applications use library operating systems (libOS) to implement their own abstractions, giving maximum control and performance.

Examples: MIT Aegis, ExOS (research).

⑤

Nanokernel

An extremely tiny kernel providing only the most minimal hardware abstraction (often just clock/interrupt handling). Almost everything else is delegated. Used mostly in research and to reduce the trusted code base.

Examples: Adeos, KeyKOS (research / embedded).

System Calls

Definition

A system call is the programming interface to the services provided by the OS. Typically written in C/C++, they are mostly accessed by programs via a high-level API rather than directly.

The three most common APIs are Win32 API (Windows), POSIX API (UNIX, Linux, macOS), and the Java API (for the JVM).

How a System Call Works

  • A number is associated with each system call; the interface maintains a table indexed by these numbers.
  • The interface invokes the intended call in the kernel and returns status and any values.
  • The caller need know nothing about the implementation — just the API. Details are hidden by the run-time support library.

For example, a C program's printf() library call internally invokes the write() system call.

③

On the Stack

Processes & File I/O System Calls

What a process is, the activities of process management, and the four low-level file I/O system calls — open(), read(), write() and close() — built on file descriptors.

The Process

Definition

A process is a program in execution — a unit of work within the system. A program is a passive entity; a process is an active entity. It needs resources (CPU, memory, I/O, files) to do its task, and termination requires reclaiming reusable resources.

Single- vs Multi-threaded

  • A single-threaded process has one program counter, executing instructions sequentially until completion.
  • A multi-threaded process has one program counter per thread.

Process Management Activities

  • Creating & deleting user and system processes.
  • Suspending & resuming processes.
  • Mechanisms for synchronization, communication and deadlock handling.

File I/O System Calls

open(), read(), write() and close() are low-level system calls used to interact with files through file descriptors, defined in unistd.h and fcntl.h.

File Descriptor (fd) A file descriptor is a small integer identifying an open file. Each process has its own descriptor table. Standard descriptors: 0 = stdin (keyboard), 1 = stdout (console), 2 = stderr.
CallSyntaxPurpose & Return
open()int fd = open(pathname, flags, mode)Open/create a file and obtain a descriptor. Returns ≥ 0 on success, −1 on failure.
read()ssize_t read(fd, buf, count)Read bytes into a buffer. Returns bytes read (>0), 0 at EOF, −1 on error.
write()ssize_t write(fd, buf, count)Send data from a buffer to a file/device/pipe/socket. Returns bytes written or −1.
close()int close(fd)Release the descriptor and free kernel resources. Returns 0 on success, −1 on error.

Common open() Flags

O_RDONLY read only O_WRONLY write only O_RDWR read & write O_APPEND write at end O_CREAT create if absent O_EXCL ensure file is new O_TRUNC truncate existing

File permissions (mode) use octal values for Owner / Group / Others — e.g. 0644 → rw- r-- r--. Each open file has a current position called the offset: reading read(fd,buffer,5) on "Hello World" returns "Hello", and the next read continues from where it left off.

Exam Tip — Return Values Remember the special return values: read() returns 0 at EOF; open() returns −1 on failure (a valid fd is otherwise ≥ 0). write(1, "Hello\n", 6) prints to stdout.

Inter-Process Communication (IPC)

Independent vs cooperating processes, why processes cooperate, and the two fundamental IPC models — shared memory and message passing (pipes & sockets).

Independent vs Cooperating Processes

Independent

  • Cannot affect or be affected by other processes.
  • Does not share data with any other process.
vs

Cooperating

  • Can affect or be affected by other processes.
  • Shares data and coordinates with others.

Why Processes Cooperate

IS

Information Sharing

Several users may want the same piece of information (e.g. a shared file).

CS

Computation Speedup

Break a task into subtasks running in parallel — only achievable with multiple cores.

M

Modularity

Divide system functions into separate processes or threads.

Cv

Convenience

An individual user works on many tasks at once — editing, music, compiling.

Two Fundamental IPC Models

Shared Memory

  • A region of memory is shared by cooperating processes.
  • Processes exchange information by reading and writing to the shared region.
  • The region resides in the address space of the creating process; others must attach it to theirs.
vs

Message Passing

  • Communication by explicitly sending and receiving messages through the OS kernel.
  • No direct access to a shared region.
  • Implemented via pipes and sockets.

  • Fastest way of IPC.
  • No kernel copying after mmap.
  • Efficient for large data.

  • No synchronization — processes must coordinate access themselves.

IPC Using Pipes

Pipe

Pipes provide one-way (half-duplex) communication between related processes, implemented on kernel I/O buffers. A pipe connects the standard output of one process to the standard input of another — the basis of UNIX shell pipelines.

  • Accessed like ordinary files — they have inodes and use file descriptors.
  • The pipe inode resides in the kernel, not the physical file system.
  • A pipe exists only while its creator process exists and has limited size (since Linux 2.6.11, capacity is 65,536 bytes).
Creating & Using Pipes int pipe(int fd[2]) opens two descriptors — fd[0] for reading, fd[1] for writing — returning 0 on success, −1 otherwise. A child inherits open descriptors from the parent, so a pipe lets a process communicate with its child. Both must close unused descriptors, then use read() and write().

The parent creates a pipe before forking. The child writes an integer into the write-end fd[1]; the parent reads it from the read-end fd[0].

pipe_demo.c
int main() { int fd[2], pid, x; pipe(fd); // create the pipe first if ((pid = fork()) == -1) { perror("fork"); exit(1); } if (pid == 0) { // CHILD — writer x = 42; write(fd[1], &x, sizeof(int)); // (descriptor, buffer, size) } else { // PARENT — reader read(fd[0], &x, sizeof(int)); printf("Parent received: %d\n", x); } return 0; }

Because the child inherits the open descriptors from the parent, both share the same kernel pipe buffer. In real code each side should close() the descriptor it does not use.

Shared-Memory IPC — the system-call flow A producer creates the segment with shm_open(), sizes it with ftruncate(), and maps it into its address space with mmap() (Linux exposes it as /dev/shm/…). The consumer shm_open()s the same name and mmap()s it, then both read/write through the returned pointer. Cleanup is munmap() then shm_unlink(). Fastest IPC (no kernel copy after mmap) but provides no synchronization — you must add a semaphore or mutex yourself.

Process Synchronization

Race conditions, the critical-section problem, the producer–consumer problem, the three requirements for a correct solution, and Peterson's software solution.

Race Conditions

When multiple processes/threads run concurrently and share memory, files or variables, accessing shared data without coordination creates a problem.

Race Condition

A situation where the final result depends on the timing/order of threads, not on the logic. For example, two concurrent counter++ operations each compile to load → add → store; if these interleave arbitrarily, the result can be 6 (wrong) instead of the expected 7.

The Critical Section

Critical Section For n processes {P₀ … Pₙ₋₁}, each has a critical section — a code segment that changes shared variables, tables or files. The rule: when one process is in its critical section, no other may be in its own. No two processes execute their critical sections simultaneously.

Each process structures its code into four sections:

Structure of a Process
1
Entry SectionRequests permission to enter the critical section.
2
Critical SectionAccess to shared resources happens here.
3
Exit SectionReleases the critical section.
4
Remainder SectionThe rest of the code.

The Producer–Consumer Problem

Classic synchronization problem

One or more producers generate data, one or more consumers use it, and both share a common buffer. When they run concurrently, two issues arise.

Producer too fast

The buffer becomes full and new data may overwrite old data.

Consumer too fast

The buffer becomes empty and the consumer reads invalid/garbage data.

Three Requirements of a Correct Solution

ME

Mutual Exclusion

If Pᵢ is in its critical section, no other process can be in theirs.

P

Progress

If no one is in the CS and some want in, only contending processes decide who enters next — and the choice cannot be postponed indefinitely.

BW

Bounded Waiting

There is a limit on how many times others enter after a process requests entry and before it is granted.

Peterson's Solution

Software-based

A software algorithm that ensures mutual exclusion between two processes (P₀ and P₁) without hardware support. It works only for two processes.

Why Peterson's solution can fail on modern systems Modern CPUs perform instruction reordering, compiler optimizations, and have cache inconsistency across cores. This reordering + cache delay can break Peterson's solution — leading to race conditions and incorrect synchronization. The fix is hardware support (next section).

Hardware Support, Locks, Semaphores & Monitors

Hardware Support for Synchronization

Hardware support provides atomic and ordered operations to safely manage shared data. Three types:

MB

Memory Barriers

Ensure memory operations are executed in order.

HI

Hardware Instructions

Special atomic CPU instructions — Test-and-Set and Compare-and-Swap (CAS).

AV

Atomic Variables

Variables supporting indivisible operations completing in one step, with no interruption.

The Concept of a Lock

Lock

A lock is a variable used to ensure that only one process/thread enters the critical section at a time.

Why a naive lock fails The two steps if (lock == 0) then lock = 1; are NOT atomic — two threads can both pass the check. We need an operation that does CHECK + SET in one atomic step: that's Test-and-Set. Compare-and-Swap compares a value and updates it only if it matches the expected value.

Mutex Locks

Mutex (Mutual Exclusion Lock)

A synchronization mechanism ensuring only one thread enters the critical section at a time. Protect a CS by first acquire()-ing the lock, then release()-ing it — both must be atomic (via hardware instructions like CAS).

Spinlock (Busy Waiting)

  • The thread keeps checking — busy-waiting — so it's called a spinlock.
  • Used for kernel internals and short critical sections.
vs

Blocking Mutex (Sleeping)

  • The thread goes to sleep and the OS wakes it later.
  • Modern mutexes (pthread) try fast → spin briefly → sleep (futex).

The pthread interface: pthread_mutex_init, pthread_mutex_lock (blocks if locked), pthread_mutex_trylock (returns "busy" immediately), pthread_mutex_unlock and pthread_mutex_destroy. Mutex variables are of type pthread_mutex_t and must be initialized before use.

Exam Tip — Lock Ordering & Deadlock If Thread 1 needs lock x then y, while Thread 2 needs lock y then x, they can deadlock. The fix is a consistent lock ordering — every thread acquires locks in the same global order.

Condition Variables

Mutex ensures mutual exclusion but does not tell a thread when to proceed — it provides no condition-based synchronization.

Condition Variable

Allows a thread to sleep until a specific condition becomes true. The producer–consumer solution uses two condition variables — one for "buffer not full", one for "buffer not empty".

Semaphores

Semaphore

A more sophisticated synchronization tool than mutex locks. A semaphore S is an integer accessed only via two indivisible (atomic) operations: wait() and signal() — originally called P() and V().

wait(S) and signal(S)

wait(S): while (S ≤ 0) busy-wait; then S−−.
signal(S): S++.

Counting vs Binary

  • Counting semaphore — value over an unrestricted domain (e.g. 3 printers available).
  • Binary semaphore — value 0 or 1, the same as a mutex lock.

The C library: sem_init(&sem, 0, value) (0 = shared between threads, value = initial count), sem_wait(&sem) and sem_post(&sem).

Common semaphore mistakes signal(mutex) … wait(mutex) (reversed), wait(mutex) … wait(mutex) (double-wait → deadlock), or omitting wait or signal entirely. These show what goes wrong when synchronization tools are used incorrectly.
Example 1 · Enforcing execution order with a semaphore

Suppose process P1 has a statement S1 and process P2 has a statement S2, and the requirement is that S1 must execute before S2. A counting semaphore synch initialised to 0 solves this:

order.c
// semaphore synch = 0; P1: P2: S1; wait(synch); signal(synch); S2;

P2 calls wait(synch) first and blocks because synch = 0. Only after P1 finishes S1 and calls signal(synch) (making synch = 1) is P2 released to run S2 — guaranteeing the order.

Example 2 · Resource counting — the 3-printer problem

Three printers p1, p2, p3 are available, and many threads want to print concurrently. The rule: at most three threads may use a printer at a time. A counting semaphore initialised to the number of resources (3) does exactly this:

printers.c
// counting semaphore printers = 3; wait(printers); // take a printer (3 → 2 → 1 → 0) // ... use the printer ... signal(printers); // release it (back up)

The first three threads pass wait() (the value drops 3 → 2 → 1 → 0). The fourth thread finds the value at 0 and blocks until one of the first three calls signal(). This is the canonical use of a counting semaphore: tracking how many units of a resource remain free.

Semaphore vs Mutex

Use a Mutex

  • Protecting shared data.
  • Ownership matters — the same thread that locks must unlock.
vs

Use a Semaphore

  • Signaling between threads.
  • Resource counting.
  • A different thread can wake another.

Monitors

Monitor

A high-level abstraction providing a convenient mechanism for process synchronization. It is an abstract data type whose internal variables are only accessible by code within its procedures, and only one process may be active inside the monitor at a time.

Threads & Multithreading

Motivation & Benefits

Most modern applications are multithreaded — separate threads handle separate tasks (update display, fetch data, spell check, answer a network request). Process creation is heavy-weight; thread creation is light-weight.

R

Responsiveness

Execution may continue even if part of the process is blocked — vital for user interfaces.

RS

Resource Sharing

Threads share the process's resources — easier than shared memory or message passing.

E

Economy

Cheaper than process creation; thread switching has lower overhead than context switching.

S

Scalability

A process can take advantage of multicore architectures.

Concurrency vs Parallelism

Concurrency

  • Supports more than one task making progress.
  • Achievable on a single core via a scheduler.
vs

Parallelism

  • The system performs more than one task simultaneously.
  • Requires multiple cores.

Two types of parallelism: data parallelism distributes subsets of the same data across cores (same operation on each), while task parallelism distributes threads across cores, each performing a unique operation.

Amdahl's Law

Amdahl's Law Identifies performance gains from adding cores when an application has both serial (S) and parallel parts. If an app is 75% parallel / 25% serial, moving from 1→2 cores gives ~1.6× speedup. As cores N → ∞, speedup approaches 1 / S. The serial portion has a disproportionate effect on the gains.
Worked Example · Applying Amdahl's Law

Formula: Speedup ≤ 1 / ( S + (1 − S) / N ), where S = serial fraction and N = number of cores.

Given an application that is 25% serial (S = 0.25, so 75% parallel):

  • N = 2 cores: Speedup = 1 / (0.25 + 0.75/2) = 1 / 0.625 = 1.6×
  • N = 4 cores: Speedup = 1 / (0.25 + 0.75/4) = 1 / 0.4375 ≈ 2.29×
  • N → ∞: Speedup → 1 / S = 1 / 0.25 = 4× (the hard ceiling)

The takeaway: no matter how many cores are added, the speedup can never exceed 1/S. Just 25% serial code caps the gain at 4×, which is why reducing the serial fraction matters more than adding cores.

User Threads vs Kernel Threads

User Threads

Managed by a user-level threads library. Three primary libraries: POSIX Pthreads, Windows threads, Java threads.

Kernel Threads

Supported directly by the kernel. Found in virtually all general-purpose OSes — Windows, Linux, macOS, iOS, Android.

The Three Multithreading Models

M:1

Many-to-One

Many user threads → one kernel thread. One blocking call blocks all; no true parallelism. Rare today (e.g. GNU Portable Threads).

1:1

One-to-One

Each user thread maps to a kernel thread — more concurrency, but overhead limits the count. Used by Windows & Linux.

M:M

Many-to-Many

Many user threads mapped to many kernel threads — the OS creates a sufficient number. The two-level model also lets a user thread bind to a kernel thread.

Pthreads & Implicit Threading

Pthreads

A POSIX standard (IEEE 1003.1c) API for thread creation and synchronization. It is a specification, not an implementation — common in UNIX, Linux and macOS. Java threads are managed by the JVM, typically using the underlying OS model; standard practice is to implement the Runnable interface.

Implicit threading moves thread creation and management to compilers and run-time libraries. Five methods: Thread Pools, Fork-Join, OpenMP, Grand Central Dispatch, and Intel Threading Building Blocks (TBB).

Thread Pools Create a number of threads in a pool that await work. Benefits: servicing a request with an existing thread is faster than creating one; the number of threads is bounded by the pool size; and tasks are decoupled from thread creation.

Threading Issues

⑂

fork() & exec()

Does fork() duplicate one thread or all? exec() usually replaces all threads.

⚑

Signal Handling

To which thread is a signal delivered? To the applicable thread, every thread, certain threads, or one assigned thread.

✕

Thread Cancellation

Terminating a target thread — asynchronous (immediate) or deferred (checked at cancellation points). Default is deferred.

TLS

Thread-Local Storage

Each thread gets its own copy of data — visible across function invocations, unique per thread.

SA

Scheduler Activations

Provide upcalls from kernel to the thread library to maintain the right number of kernel threads, via lightweight processes (LWPs).

L

Linux Tasks

Linux calls them tasks; creation uses clone(), whose flags control how much the child shares with the parent.

Memory Management

Background and protection, address binding, logical vs physical addresses, the MMU, dynamic loading & linking, and the contiguous allocation schemes — fixed and dynamic partitioning with their fit algorithms.

Background

  • A program must be brought from disk into memory and placed within a process to run.
  • Register access takes one CPU clock; main memory can take many cycles; a cache sits between them.
  • Protection of memory is required for correct operation.
Base & Limit Registers

A process must access only addresses in its own address space. A pair of base and limit registers defines its logical address space; the CPU checks every user-mode access against them. The instructions that load these registers are privileged.

Early Memory Management Approaches

1

Bare Machine

No operating system at all. Used in dedicated systems (e.g. a computer monitoring a chemical plant). Memory is allocated to each task by a fixed, predefined policy.

2

Resident Monitor

Memory is divided into two partitions — one reserved for OS services, the other (user space) for user processes. A fence address separates them so a user program cannot overwrite the monitor.

3

Swapping

A process can be swapped out of memory to a backing store and later swapped back in to continue, allowing the total memory used by all processes to exceed physical memory.

Address Binding

Addresses are represented differently at different stages of a program's life — source code uses symbolic addresses, compiled code uses relocatable addresses, and the linker/loader binds them to absolute addresses. Binding can happen at three stages:

①

Compile Time

If the location is known in advance, absolute code is generated — but the code must be recompiled if the start location changes.

②

Load Time

If the location is unknown at compile time, relocatable code is generated.

③

Execution Time

Binding is delayed until run time so the process can be moved during execution — needs hardware support (base/limit registers).

Logical vs Physical Address Space

Logical (Virtual) Address

  • Generated by the CPU.
  • The set of all such addresses is the logical address space.
→

Physical Address

  • The address seen by the memory unit.
  • They are the same under compile/load-time binding, but differ under execution-time binding.
Memory-Management Unit (MMU) A hardware device that maps virtual → physical addresses at run time. In the simple relocation-register scheme, the register's value is added to every address a user process generates. The user program never sees real physical addresses.

Static, Dynamic Loading & Dynamic Linking

Static linking combines the system libraries and program code into a single binary image at compile/load time — simple, but every program carries its own copy of the libraries. The two dynamic techniques below defer work to run time.

Dynamic Loading

A routine is not loaded until it is called — better memory utilization, since unused routines are never loaded. Useful for large, infrequently-used code. No special OS support required.

Dynamic Linking

Linking is postponed until execution time. A small stub locates the memory-resident library routine and replaces itself with the routine's address. Also known as shared libraries. (Contrast: static linking binds the libraries in at build time.)

Memory Management Techniques

Two Families of Allocation

Contiguous

The executing process must be loaded entirely in main memory.

  • Fixed (static) partitioning
  • Variable (dynamic) partitioning

Non-Contiguous

The process is loaded into non-contiguous locations.

  • Paging
  • Segmentation

Fixed (Static) Partitioning

  • The number of partitions is fixed; sizes may or may not be equal.
  • Being contiguous, no spanning is allowed.
  • The number of partitions defines the degree of multiprogramming.
  • Suffers from internal fragmentation.

Partition Allocation (Fit) Algorithms

1st

First Fit

Allocate the first sufficient block from the top of memory.

B

Best Fit

Allocate the smallest sufficient free partition.

W

Worst Fit

Allocate the largest sufficient free partition.

N

Next Fit

Like first fit, but searches from the last allocation point.

Dynamic (Variable) Partitioning

Partitions are created at run-time according to each process's need — the partition size equals the incoming process, so the number of partitions is not fixed.

Advantages

  • No internal fragmentation — space allocated exactly to need.
  • No restriction on the degree of multiprogramming.
  • No limit on process size (vs largest fixed partition).

Disadvantages

  • Difficult to implement — allocation at run-time.
  • External fragmentation, requiring a compaction algorithm.

Paging & Segmentation

Non-contiguous allocation — paging with its address-translation scheme, the page table and TLB, and segmentation with its segment table, plus a side-by-side comparison.

Paging

Paging

Allows a process's logical address space to be non-contiguous. Physical memory is divided into fixed-size frames (a power of 2, between 512 B and 8 KB); logical memory into same-size pages. To run a program of n pages, find n free frames and set up a page table to translate logical to physical addresses. Internal fragmentation is possible.

Address-Translation Scheme

Page Number + Page Offset The CPU-generated address splits into a page number (p) — an index into the page table holding each page's base address — and a page offset (d), combined with the base to give the physical address. For a logical space of 2ᵐ and page size 2ⁿ, p uses (m−n) bits and d uses n bits.

Implementing the Page Table

  • The page table is kept in main memory; PTBR points to it and PTLR indicates its length.
  • Naively, every access needs two memory accesses — one for the table, one for the data.
  • This is solved by the TLB (Translation Look-aside Buffer) — a fast associative-memory cache.
TLB

On a TLB miss, the page table is consulted and the <page, frame> pair is added to the TLB (replacing an entry if full). Some entries (kernel code) are wired down. ASIDs (address-space identifiers) uniquely identify each process so the TLB need not be flushed on every context switch.

Segmentation

Segmentation

A memory-management scheme that supports the user's view of memory. A program is a collection of segments — logical units such as the main program, procedures, functions, objects, local & global variables, stack, symbol table and arrays.

A logical address is a tuple <segment-number, offset>. The segment table maps it: each entry has a base (starting physical address) and a limit (segment length). STBR points to the table; STLR gives the number of segments — segment s is legal if s < STLR.

Advantages of Segmentation

  • No internal fragmentation.
  • Segment tables consume less memory than page tables (one entry per segment).
  • Lends itself to sharing data among processes.
  • Each segment can be set its own access right.

Disadvantages

  • External fragmentation.
  • Costly memory-management algorithm.
  • Unequal segment sizes are awkward for swapping.
Exam Tip — Paging vs Segmentation Paging = fixed-size pages, invisible to the user, suffers internal fragmentation. Segmentation = variable-size logical units matching the user's view, suffers external fragmentation. Both are non-contiguous.

Virtual Memory & Demand Paging

The idea of virtual memory, demand paging with the lazy swapper, the valid–invalid bit, and the step-by-step handling of a page fault.

What is Virtual Memory?

Virtual Memory

The separation of user logical memory from physical memory. Only part of a program needs to be in memory to execute, so the logical address space can be much larger than physical memory.

  • Only part of the program needs to be in memory — less memory needed.
  • Faster response and support for more users.
  • Allows address spaces to be shared by several processes.
  • Allows for more efficient process creation.

Virtual memory can be implemented via demand paging or demand segmentation.

Demand Paging

The Lazy Swapper Demand paging brings a page into memory only when it is needed. A lazy swapper never swaps a page in unless it will be needed — a swapper that deals with pages is called a pager.

The Valid–Invalid Bit

Valid–Invalid Bit

Each page-table entry has a valid–invalid bit: v = in-memory, i = not-in-memory. Initially all entries are set to i. During address translation, if the bit is i, a page fault traps to the OS.

Page Replacement & Thrashing

Basic page replacement, the three classic algorithms — FIFO, Optimal and LRU — with their page-fault behaviour, plus thrashing.

Basic Page Replacement

Replacing a Victim Frame
1
Locate the PageFind the desired page on disk.
2
Find a FrameIf free, use it; otherwise use a replacement algorithm to select a victim frame.
3
Bring In the PageLoad the desired page into the newly free frame; update the page and frame tables.
4

We want the lowest page-fault rate. Algorithms are evaluated by running them on a reference string of memory references and counting page faults. The standard example string is 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5.

The Three Classic Algorithms

FIFO

First-In-First-Out

Replaces the oldest page. Simple but can perform poorly — and exhibits Belady's anomaly (more frames can mean more faults). On the example: 9 faults with 3 frames, 10 faults with 4.

OPT

Optimal

Replaces the page not used for the longest future period. Gives the fewest faults (6 on the example) but requires future knowledge — used mainly as a benchmark.

LRU

Least Recently Used

Replaces the page not used for the longest past period — an approximation of Optimal. Implemented with a counter per page entry that records the clock at each reference.

Exam Tip — Page-Fault Counts Memorise the example string 1,2,3,4,1,2,5,1,2,3,4,5: with the same frames, OPT ≤ LRU ≤ FIFO in faults. OPT looks forward; LRU looks backward; FIFO ignores usage. Watch for Belady's anomaly in FIFO.

Thrashing

Thrashing

When a process does not have enough frames, it spends more time paging than executing — page faults follow one another so closely that the CPU utilization collapses. The OS may increase the degree of multiprogramming and make it worse; the working-set model helps allocate enough frames to avoid it.

Deadlocks

The four necessary conditions, the four ways of handling deadlocks — prevention, avoidance (Banker's Algorithm), detection and recovery.

What is a Deadlock?

Deadlock

A set of processes is deadlocked when every process in the set is waiting for an event that only another process in the set can cause — typically the release of a resource. None can proceed.

The Four Necessary Conditions

A deadlock can arise only if all four hold simultaneously:

①

Mutual Exclusion

At least one resource is held in a non-sharable mode.

②

Hold and Wait

A process holding resources is waiting to acquire more held by others.

③

No Preemption

Resources cannot be forcibly taken — only released voluntarily.

④

Circular Wait

A circular chain of processes exists, each waiting for a resource held by the next.

Four Ways to Handle Deadlocks

StrategyIdea
PreventionEnsure at least one of the four necessary conditions cannot hold — e.g. eliminate hold-and-wait or impose a total ordering to break circular wait.
AvoidanceUse advance information about resource needs to keep the system in a safe state — the Banker's Algorithm.
DetectionAllow deadlocks to occur, then run a detection algorithm (a wait-for graph) to find them.

Disk & Storage Management

Disk geometry, the components of disk access time, and the five disk-scheduling algorithms — FCFS, SSTF, SCAN, C-SCAN and LOOK — with a worked access-time example.

Disk Structure

Magnetic disks provide the bulk of secondary storage. Data is stored on a stack of rotating magnetic platters in concentric tracks; read/write heads move radially to position over the desired track.

S

Sector

The smallest unit that can be read or written — data is stored in equal-size blocks (512 to 4096 bytes).

T

Track

A circle of consecutive sectors on one surface, along one revolution.

C

Cylinder

A collection of tracks at the same radius across multiple surfaces — accessing within a cylinder needs only a faster head switch, not head movement.

Disk Access Time

Access Time = Seek + Rotational Latency + Transfer Seek time — time to move the head to the desired track. Rotational latency — time for the desired sector to rotate under the head (average = ½ rotation). Transfer time = transfer size ÷ transfer rate.
Worked Example

Given transfer size 8 KB, average seek 12 ms, 7200 RPM, transfer rate 4 MB/s:
Rotational latency = (60 / 7200) ÷ 2 = 8.33 ÷ 2 = 4.16 ms.
Transfer time = 8 / 4000 = 0.002 s = 2 ms.
Total access time = 12 + 4.16 + 2 = 18.16 ms.

The OS schedules disk I/O requests in a good order to improve access time and bandwidth (total bytes transferred ÷ total time). Using the standard queue 98, 183, 37, 122, 14, 124, 65, 67 with the head at cylinder 53:

Topic 2 · Zombie & Orphan Processes

Both arise from the parent–child relationship created by fork(). When a child finishes, the kernel keeps a small entry until the parent collects its exit status with wait(). What happens around that step defines the two cases.

Z

Zombie Process

A child that has terminated but whose parent has not yet called wait(). The process is dead, yet its entry lingers in the process table (state Z / defunct) holding its exit status. Too many zombies waste PIDs.

O

Orphan Process

A child whose parent terminated first, while the child is still running. The orphan is adopted by init (PID 1) (or a subreaper), which calls wait() on its behalf so it never becomes a zombie.

Code · Creating a Zombie process

The child exits immediately, but the parent sleeps without calling wait() — so the finished child stays a zombie during the sleep.

zombie.c
#include <stdio.h> #include <stdlib.h> #include <unistd.h> int main() { pid_t pid = fork(); if (pid > 0) { // PARENT sleep(30); // not calling wait() yet // during these 30s the child is a ZOMBIE } else if (pid == 0) { // CHILD exit(0); // child finishes immediately } return 0; }

How to observe: while the parent sleeps, run ps -el | grep defunct — the child shows state Z. Fix: have the parent call wait() (or handle the SIGCHLD signal) to reap the child.

Code · Creating an Orphan process

The parent exits first; the child keeps running and is re-parented to init.

orphan.c
#include <stdio.h> #include <unistd.h> int main() { pid_t pid = fork(); if (pid > 0) { // PARENT exit(0); // parent finishes first } else if (pid == 0) { // CHILD — now an ORPHAN sleep(10); printf("My new parent PID = %d\n", getppid()); // getppid() now returns 1 (init), not the old parent } return 0; }

After the parent exits, getppid() in the child returns 1 — it has been adopted by init, which will reap it when it finishes.

One-line difference Zombie = child dead, parent alive but hasn't reaped it (harmful — leaks PIDs). Orphan = parent dead, child alive, re-parented to init (harmless — init reaps it).

Topic 7 · Deadlock Algorithms — Banker's & Detection

Two matrix-based algorithms appear in the paper. Banker's is for avoidance (decide before granting, using each process's maximum claim). Detection is run after allocation (using each process's current request) to find an existing deadlock.

1

Banker's Algorithm (Avoidance)

Needs Max, Allocation, Available → computes Need = Max − Allocation. Grants a request only if the resulting state is safe (a safe sequence exists). Uses the Safety + Resource-Request sub-algorithms.

2

Detection Algorithm

Needs Allocation, Request, Available. Repeatedly finds a process whose Request ≤ Work, finishes it and reclaims its resources. Any process left unfinished is deadlocked.

10 Marks D1. What are threads? How are they implemented and how do they work?
A thread is the basic unit of CPU utilisation — a lightweight unit of execution inside a process. It consists of a thread ID, program counter, register set and a stack, while it shares the code section, data section and OS resources (open files, signals) with the other threads of the same process.
Single-threaded vs Multithreaded

A traditional (heavyweight) process has a single thread of control. A multithreaded process has several threads running in the same address space, so they can do several things at once — e.g. a web browser rendering a page while downloading images, or a server handling many clients.

Why threads (benefits)
  • Responsiveness — part of the program can keep running even if another part is blocked.
  • Resource sharing — threads share memory by default, so no extra IPC is needed.
  • Economy — creating and switching threads is far cheaper than processes (no full address-space copy).
  • Scalability — threads run in parallel on multiple cores.
How threads are implemented

Threads exist at two levels, linked by a multithreading model:

  • User-level threads — managed by a user-space thread library without the kernel's knowledge. Fast to create and switch, but if one thread makes a blocking system call the whole process blocks.
  • Kernel-level threads — created and scheduled by the OS. Slower to manage, but one thread blocking does not stop the others, and they can run on different CPUs.

The three mapping models are: Many-to-One (many user threads → one kernel thread), One-to-One (each user thread → its own kernel thread, used by Windows and Linux), and Many-to-Many (many user threads multiplexed onto a smaller or equal number of kernel threads). Programmers use a thread library such as Pthreads, Windows threads or Java threads.

How a thread works (lifecycle)

When a thread is created it is placed in the ready state. The scheduler dispatches it to running; it may move to blocked/waiting on I/O or a lock, return to ready when the event completes, and finally terminate. A context switch between two threads of the same process is cheap because the shared memory map does not change — only the PC, registers and stack pointer are swapped.

Conclusion: threads give concurrency within a single process at low cost; the OS realises them through user libraries, kernel threads, or a mapping between the two.

10 Marks D2. Explain the kernel and its types. Explain the monolithic kernel in detail.
The kernel is the core of the operating system — the one program that runs at all times, loaded first at boot, resident in memory, running in privileged kernel mode. It is the bridge between user applications and the hardware, and it manages processes, memory, devices and the system-call interface.
Types of kernel
  • Monolithic — the entire OS runs in kernel space as one program.
  • Microkernel — only the essentials run in the kernel; other services run as user-space servers.
  • Hybrid — microkernel structure with performance-critical services kept in the kernel (Windows, macOS).
  • Exokernel — the kernel only allocates hardware; abstractions live in library OSes.
  • Nanokernel — an extremely small kernel providing minimal hardware abstraction.
Monolithic kernel — in detail

In a monolithic kernel, the entire operating system — scheduler, memory manager, file system, device drivers, IPC and networking — runs together in a single address space in kernel mode. The modules are not isolated; they call one another directly as ordinary function calls.

  • High performance — because services communicate by direct calls, there is no message-passing or mode-switch overhead between them. This is the biggest advantage.
  • Less secure / less stable — since everything shares one space, a bug or crash in any component (e.g. a faulty driver) can bring down the whole system.
  • Large and harder to maintain — the code base is big and tightly coupled.
  • Extensible via modules — modern monolithic kernels (like Linux) use loadable kernel modules so drivers can be added at run time without rebuilding the kernel.

Examples: UNIX, Linux, MS-DOS. Structure in words: [user applications] → system-call interface → [one large kernel block: scheduler + memory + file system + drivers + IPC] → [hardware].

Conclusion: the monolithic design trades isolation for speed — fast because of direct calls, but fragile because a single fault can crash the OS. Microkernels invert that trade-off.

10 Marks D3. What is synchronization and how is it achieved?
Process synchronization is the coordination of concurrent processes or threads that share data, so that they execute in a controlled order and the shared data stays consistent.
Why it is needed — the race condition

When several processes access shared data at the same time and the final result depends on the order in which their instructions interleave, we have a race condition. For example, counter++ is really load → increment → store; if two threads interleave these steps, one update can be lost. The region of code that touches shared data is the critical section, and the goal of synchronization is to control entry into it.

Requirements of a correct solution
  • Mutual exclusion — only one process in its critical section at a time.
  • Progress — if no one is in the critical section, a waiting process must be allowed in without indefinite delay.
  • Bounded waiting — there is a limit on how long a process waits to enter.
How synchronization is achieved
  • Software solutions — e.g. Peterson's algorithm for two processes, using flag[] and turn variables.
  • Hardware support — atomic instructions Test-and-Set and Compare-and-Swap (CAS) perform check-and-set in one indivisible step (software locks alone fail because the check and the set are separate steps). Memory barriers stop instruction reordering.
  • Mutex lock — a boolean lock acquired before and released after the critical section; only the owning thread may unlock it.
  • Semaphores — an integer with atomic wait() and signal(). A binary semaphore acts as a lock; a counting semaphore tracks a resource with several units. A negative value tells how many processes are blocked.
  • Monitors — a high-level construct that bundles shared data with the procedures that operate on it, automatically guaranteeing mutual exclusion, plus condition variables (wait/signal) for ordering.

Conclusion: synchronization prevents race conditions by enforcing mutual exclusion over the critical section, achieved through atomic hardware instructions and the higher-level tools (mutexes, semaphores, monitors) built on top of them.

10 Marks D4. Explain sockets, pipes and shared memory — how each is created, how each works, the differences, which is best and why, and why we use them.
Sockets, pipes and shared memory are three inter-process communication (IPC) mechanisms. We use them because cooperating processes need to share information, divide work (modularity) and speed up computation, and a process cannot read another process's private memory directly.
1. Shared Memory

How it works: a region of physical memory is mapped into the address space of two or more processes; they then communicate by simply reading and writing that region like normal variables. The kernel is involved only when the region is set up — after that, access is at full memory speed.

How it is created (POSIX): shm_open() creates/opens a named segment → ftruncate() sets its size → mmap() maps it in and returns a pointer → both processes read/write through the pointer → munmap() then shm_unlink() to clean up.

Note: shared memory provides no synchronization — the programmer must add a semaphore or mutex to avoid race conditions.

2. Pipes

How it works: a pipe is a one-way (half-duplex) channel in the kernel. Data written to one end is read from the other in FIFO order. An ordinary pipe connects related processes (parent–child); a named pipe (FIFO) can connect unrelated processes.

How it is created: int pipe(int fd[2]) opens two descriptors — fd[0] for reading, fd[1] for writing. The parent calls pipe(), then fork(); the child inherits the descriptors, so the two communicate using read() and write(). Each side closes the end it does not use.

3. Sockets

How it works: a socket is an endpoint for communication, usually across a network (or between processes on the same host). Two processes each create a socket and exchange a stream or datagrams through it.

How it is created: socket() creates the endpoint → server does bind() (assign address) → listen() (mark passive) → accept() (take a connection); client does connect(). Then both use send()/recv() (or read/write).

Which is best, and why

There is no single best — it depends on the need:

  • Shared memory is best for speed and large data on one machine, because after mapping there is no kernel copying — but you must add your own synchronization.
  • Pipes are best for simple, one-way data flow between related processes — naturally synchronized and easy to use.
  • Sockets are best (and the only choice) when processes are on different machines, and for two-way client–server communication.

Conclusion: use shared memory for fast local bulk transfer (with a semaphore), pipes for simple parent–child streams, and sockets for networked or full-duplex communication.

5 Marks D5. How does one process communicate with another process?
Processes communicate through Inter-Process Communication (IPC). Because each process has its own private address space, they cannot read each other's memory directly — the OS provides IPC mechanisms instead.
Two fundamental models
  • Shared memory — the OS sets up a region both processes can map; they then communicate by reading/writing it directly. Fast, but the programmer must synchronize access.
  • Message passing — processes exchange messages through the kernel using send() and receive(). No shared address space is needed; the kernel handles each message. Easier to use and works across machines, but slower.
Common mechanisms

Built on these models are pipes (one-way, related processes), named pipes/FIFOs (unrelated processes), message queues, and sockets (across a network). The choice depends on whether the processes are on the same machine and whether speed or simplicity matters more.

In short: one process talks to another through IPC — either by sharing a memory region or by passing messages via the kernel.

10 Marks D7. Explain the Banker's Algorithm.
The Banker's Algorithm is a deadlock-avoidance algorithm. Before granting a resource request it checks whether doing so would leave the system in a safe state; if not, the request is made to wait. It is named after a banker who only lends money if he can still satisfy all customers.
Data structures (n processes, m resource types)
  • Available[m] — free instances of each resource.
  • Max[n][m] — maximum demand of each process.
  • Allocation[n][m] — currently held by each process.
  • Need[n][m] = Max − Allocation — what each process may still request.
Safe state

A state is safe if there is a safe sequence — an order in which every process can finish, because each one's remaining Need can be met from the currently Available resources plus those released by the processes before it.

Safety algorithm
  • Set Work = Available and Finish[i] = false for all i.
  • Find a process i with Finish[i] = false and Needi ≤ Work; pretend it finishes: Work = Work + Allocationi, Finish[i] = true.
  • Repeat. If every process finishes, the state is safe.
Resource-request algorithm

When process Pi requests resources: check Request ≤ Need and Request ≤ Available; if so, pretend to allocate and run the safety check. If the result is safe, grant it; otherwise Pi waits.

Worked example

For Available = (3,3,2) with the standard 5-process/3-resource table, the safety algorithm finds the safe sequence <P1, P3, P4, P0, P2>, so the system is safe. (Full table and step-by-step working are in the Banker's numerical above.)

Conclusion: the Banker's Algorithm never enters an unsafe state, so deadlock is avoided — but it needs each process's maximum demand in advance.

10 Marks D8. ★ Very important — Explain interrupt handling.
An interrupt is a signal sent to the CPU by an I/O device (or by software) indicating that an event has occurred and needs immediate attention. Interrupt handling is the mechanism by which the CPU temporarily suspends its current work, services the event, and then resumes. The operating system is fundamentally interrupt-driven.
Types of interrupts
  • Hardware interrupt — raised by an I/O device (e.g. keyboard, disk, timer) when it finishes an operation.
  • Software interrupt / trap (exception) — generated by the running program, caused by an error (e.g. division by zero, invalid memory access) or a deliberate user request (a system call).
Interrupt handling steps
  1. The CPU finishes the current instruction (it checks for interrupts between instructions).
  2. The CPU saves the current state — the program counter and CPU registers — so it can resume later. The interrupt architecture must save the program counter.
  3. Control is transferred to the interrupt handler. The CPU uses the interrupt vector table — an array indexed by interrupt number that holds the address of each handler — to jump to the right routine.
  4. The OS identifies the type of interrupt.
  5. The corresponding Interrupt Service Routine (ISR) executes and services the device.
  6. The CPU restores the saved state (program counter and registers).
  7. Execution resumes from exactly where it was interrupted.
Important points
  • Each device controller has a local buffer and communicates with the CPU through interrupts.
  • The CPU checks for pending interrupts between instructions.
  • Interrupts can be masked (disabled) temporarily during critical operations, and have priorities so urgent events are served first.
  • A trap is just a software-generated interrupt — the same handling machinery is used.
Interrupt-driven I/O cycle (in words)

The device driver initiates I/O → the CPU keeps executing, checking for interrupts between instructions → when the device has input ready / output complete / an error, it raises an interrupt signal → the CPU receives it and transfers control to the interrupt handler → the handler processes the data and returns from the interrupt → the CPU resumes the interrupted task.

Conclusion: interrupt handling lets the CPU respond immediately to events without constantly polling devices — it saves state, runs the ISR via the interrupt vector, and restores state, which is why it is the backbone of an interrupt-driven OS.

5 Marks Q1. What is an Operating System? State its main goals and functions.
An Operating System (OS) is a program that acts as an intermediary between the user of a computer and the computer hardware. It controls and coordinates the use of hardware among various application programs for the different users.
Main Goals
  • Execute user programs and make solving user problems easier.
  • Make the computer system convenient to use.
  • Use the computer hardware in an efficient manner (good resource utilization).
Major Functions
  • Process management — creating, scheduling, and terminating processes.
  • Memory management — allocating and de-allocating main memory.
  • File management — organizing, storing, and protecting files.
  • Device (I/O) management — controlling devices through drivers.
  • Security & protection — controlling access to system resources.

Conclusion: The OS is therefore both a resource allocator and a control program that bridges users and hardware.

10 Marks Q2. Explain the various types of Operating Systems with their characteristics.

An operating system can be classified based on how it manages jobs, users, and time. The major types are described below.

1. Batch Operating System

Similar jobs are grouped into batches and executed together without user interaction. A resident monitor controls execution. Advantage: high throughput. Drawback: no interaction, CPU often idle during I/O, and a difficult-to-debug environment.

2. Multiprogramming Operating System

Several jobs are kept in main memory at once; when one job waits for I/O, the CPU switches to another job. This keeps the CPU busy at all times and improves utilization.

3. Time-Sharing (Multitasking) Operating System

The CPU is shared among many users by allocating a small time quantum to each. The rapid switching gives every user the illusion of a dedicated machine. It provides quick response time (e.g., UNIX).

4. Multiprocessing Operating System

Uses two or more CPUs within a single system. It increases throughput and reliability — if one processor fails, others continue. Types: symmetric (SMP) and asymmetric multiprocessing.

5. Real-Time Operating System (RTOS)

Has well-defined, fixed time constraints that must be met. Used in embedded and control systems. Two kinds: Hard real-time (strict deadlines, e.g., airbag control) and Soft real-time (deadlines preferred, e.g., multimedia streaming).

6. Distributed Operating System

Manages a collection of independent, networked computers and presents them to the user as a single coherent system. Offers resource sharing, reliability, and computation speed-up.

10 Marks Q3. What are System Calls? Explain their types with examples.
A system call is a programmatic way for a program to request a service from the kernel of the operating system. It provides the interface between a running program and the operating system.
Need for System Calls

User programs run in user mode and cannot directly access hardware or protected resources. To perform privileged operations (reading a file, creating a process), the program issues a system call, which switches the CPU to kernel mode, executes the service, and returns control.

Types of System Calls
  • Process Control — create/terminate process, load, execute, wait, allocate memory. Examples: fork(), exit(), wait().
  • File Management — create, open, read, write, close, delete files. Examples: open(), read(), write(), close().
  • Device Management — request, release, read/write to devices. Examples: ioctl(), read(), write().
  • Information Maintenance — get/set system data, time, date, attributes. Examples: getpid(), alarm(), sleep().
  • Communication — create/delete connections, send/receive messages. Examples: pipe(), shmget(), send(), recv().
  • Protection — control access to resources, set permissions. Examples: chmod(), umask(), chown().

Conclusion: System calls form the secure, controlled boundary through which all user requests for OS services pass.

5 Marks Q3b. What is a Kernel? List its main functions.
The kernel is the central core of an operating system that is loaded first at boot, stays resident in memory at all times, runs in privileged kernel mode, and acts as the bridge between application software and hardware.
Main Functions
  • Process management — creation, scheduling, context switching, termination.
  • Memory management — allocation/de-allocation, address translation, virtual memory.
  • Device management — controlling hardware via device drivers and interrupt handling.
  • File management — organizing and controlling access to files.
  • System calls & IPC — providing the interface for user programs to request services.

In short: the kernel is the privileged manager that controls every hardware resource on behalf of all programs.

10 Marks Q3c. Explain the different types of Kernel with diagrams, and compare Monolithic vs Microkernel.
A kernel is the core of the OS. Kernels are classified by how many OS services run in kernel space versus user space. The trade-off is: more in kernel space = faster but less reliable; more in user space = more reliable but slower.
1. Monolithic Kernel

The entire OS (scheduler, memory manager, file system, drivers, IPC) runs in kernel space as one large program; modules call each other directly. Fast but a single bug can crash the whole system. Examples: Linux, UNIX, MS-DOS.

2. Microkernel

Only the essentials (IPC, basic scheduling, low-level memory) run in kernel space; other services run as user-space servers communicating via message passing. Reliable and modular but slower. Examples: QNX, MINIX, L4, Mach.

3. Hybrid Kernel

A compromise — microkernel-like structure but performance-critical services are moved back into kernel space. Examples: Windows NT, macOS (XNU).

4. Exokernel

The kernel only securely allocates hardware resources; abstractions are implemented by application-level library OSes. Maximum control. Examples: Aegis, ExOS.

5. Nanokernel

An extremely small kernel providing only minimal hardware abstraction (e.g. interrupt/clock handling). Examples: Adeos, KeyKOS.

10 Marks Q5. Explain the Process State Diagram and the contents of the Process Control Block (PCB).

As a process executes, it changes its state. The state defines the current activity of the process.

Process States
  • New — the process is being created.
  • Ready — the process is waiting to be assigned to a processor.
  • Running — instructions are being executed on the CPU.
  • Waiting (Blocked) — the process is waiting for some event (such as I/O completion).
  • Terminated — the process has finished execution.
State Transition Diagram (in words)

New → admitted → Ready → scheduler dispatch → Running. From Running a process can go to: Ready (on interrupt / time-out), Waiting (on I/O or event wait), or Terminated (on exit). From Waiting it returns to Ready when the I/O or event completes.

Process Control Block (PCB)

Each process is represented in the OS by a PCB — a data structure holding all information about the process:

  • Process state — new, ready, running, etc.
  • Process ID (PID) — unique identifier.
  • Program counter — address of the next instruction.
  • CPU registers — accumulators, index registers, stack pointers.
  • CPU scheduling information — priority, pointers to scheduling queues.
  • Memory-management information — base/limit registers, page tables.
  • Accounting information — CPU used, time limits, process numbers.
  • I/O status information — open files, allocated devices.

Conclusion: The PCB is the heart of process management — during a context switch, the OS saves the state of the old process and loads the state of the next from their PCBs.

10 Marks Q6. What is Inter-Process Communication (IPC)? Compare Shared Memory and Message Passing models.
Inter-Process Communication (IPC) is a mechanism that allows cooperating processes to exchange data and synchronize their actions.

Processes may be independent (cannot affect others) or cooperating (can affect or be affected by others). Cooperating processes need IPC for information sharing, computation speed-up, modularity, and convenience. There are two fundamental models.

1. Shared Memory Model

A region of memory is shared by cooperating processes. They exchange information by reading and writing to this shared region. The OS is involved only in setting up the shared region; thereafter communication is at memory speed.

  • Advantage: Fast — no kernel intervention per message.
  • Drawback: The programmer must provide synchronization (e.g., to avoid race conditions).
2. Message Passing Model

Processes communicate by exchanging messages through the kernel using send(message) and receive(message) primitives. No shared address space is required.

  • Advantage: Useful for distributed systems; easier to synchronize.
  • Drawback: Slower — each message involves system calls and kernel intervention.

10 Marks Q7. Explain the Critical Section Problem and the three requirements for its solution. Describe Peterson's Solution.
The critical section is the segment of code in which a process accesses shared resources (variables, files). The critical section problem is to design a protocol so that no two processes execute in their critical sections at the same time.
Structure of a Process

Each process has: Entry section (request permission) → Critical section → Exit section (release) → Remainder section.

Three Requirements of a Solution
  • Mutual Exclusion — if a process is executing in its critical section, no other process may execute in its critical section.
  • Progress — if no process is in its critical section, only processes not in their remainder section participate in deciding who enters next, and this cannot be postponed indefinitely.
  • Bounded Waiting — there is a limit on the number of times other processes can enter their critical sections after a process has requested entry and before that request is granted.
Peterson's Solution (for two processes)

Uses two shared variables: int turn; and boolean flag[2];. flag[i] = true means process Pi wants to enter; turn indicates whose turn it is.

For process Pi:

  • Entry: set flag[i] = true; set turn = j; wait while flag[j] && turn == j.
  • Critical section executes.
  • Exit: set flag[i] = false.

This satisfies all three conditions: mutual exclusion holds because both flag values cannot let both in; progress and bounded waiting are guaranteed by the turn variable.

Conclusion: Peterson's solution is a classic software-based solution proving correct synchronization is possible, though modern systems use hardware atomic instructions and semaphores.

10 Marks Q8. What is a Semaphore? Explain its types and operations. How does it solve the Producer–Consumer problem?
A semaphore is an integer variable, accessed only through two atomic operations wait() (P) and signal() (V), used to solve synchronization problems and control access to shared resources.
Operations
  • wait(S): while (S <= 0) ; S--; — decrements; blocks if value is 0.
  • signal(S): S++; — increments and possibly wakes a waiting process.
Types of Semaphores
  • Binary Semaphore (Mutex) — value is 0 or 1; used for mutual exclusion (acts like a lock).
  • Counting Semaphore — value ranges over an unrestricted domain; used to control access to a resource with multiple instances.
Producer–Consumer Solution

A bounded buffer of n slots is shared. Three semaphores are used:

  • mutex = 1 — mutual exclusion for buffer access.
  • empty = n — counts empty slots.
  • full = 0 — counts full slots.

Producer: wait(empty); wait(mutex); add item; signal(mutex); signal(full);

Consumer: wait(full); wait(mutex); remove item; signal(mutex); signal(empty);

The empty and full semaphores synchronize producer and consumer speeds, while mutex guarantees only one accesses the buffer at a time, preventing race conditions.

Conclusion: Semaphores elegantly handle both mutual exclusion and process synchronization, though careless ordering of wait/signal can cause deadlock.

Unit I · CPU Scheduling
10 Marks Q10. Explain the various CPU Scheduling Algorithms with their advantages and disadvantages.

CPU scheduling decides which process in the ready queue is allocated the CPU. The goal is to optimize criteria such as CPU utilization, throughput, turnaround time, waiting time, and response time.

1. First-Come, First-Served (FCFS)

Processes are executed in the order they arrive (FIFO queue). Non-preemptive. Simple to implement but suffers from the convoy effect — short processes wait behind long ones, giving high average waiting time.

2. Shortest Job First (SJF)

The process with the smallest next CPU burst is selected first. It is provably optimal for minimum average waiting time. Drawback: requires knowing burst length in advance and may cause starvation of long processes.

3. Shortest Remaining Time First (SRTF)

The preemptive version of SJF. If a new process arrives with a shorter remaining burst, the running process is preempted. Lower average waiting time but more context switches.

4. Priority Scheduling

The CPU is allocated to the process with the highest priority. Can be preemptive or non-preemptive. Drawback: starvation of low-priority processes, solved using ageing (gradually increasing priority of waiting processes).

5. Round Robin (RR)

Each process gets a small time quantum; after it expires the process is preempted and placed at the tail of the ready queue. Best for time-sharing. Performance depends heavily on quantum size — too large behaves like FCFS, too small causes excessive context-switch overhead.

6. Multilevel Queue & Multilevel Feedback Queue

The ready queue is partitioned into several queues (e.g., system, interactive, batch) each with its own algorithm. The feedback variant lets processes move between queues based on behaviour, balancing response and throughput.

AlgorithmPreemptive?Main Issue
FCFSNoConvoy effect
SJFNoStarvation, needs burst time
PriorityEitherStarvation (use ageing)
Round RobinYesQuantum size tuning

Conclusion: No single algorithm is best for all cases; real systems combine them (e.g., multilevel feedback queues) to balance fairness, response time, and throughput.

10 Marks Q13. Explain Paging in detail with address translation. How does the TLB improve performance?
Paging is a non-contiguous memory-management scheme that divides logical memory into fixed-size blocks called pages and physical memory into blocks of the same size called frames.
Working

When a process is loaded, its pages are placed into any available frames. A page table (one per process) maps each page number to its frame number.

Address Translation

Every logical address generated by the CPU is divided into two parts:

  • Page number (p) — index into the page table.
  • Page offset (d) — displacement within the page.

Translation: the page number p indexes the page table to obtain the frame number f; the physical address is then (f × page size) + d. This separation removes external fragmentation.

Translation Look-aside Buffer (TLB)

Accessing the page table in memory for every reference would require two memory accesses (one for the table, one for the data), doubling access time. The TLB is a small, fast, associative cache that stores recently used page-to-frame mappings.

  • On a TLB hit, the frame number is obtained immediately — only one memory access.
  • On a TLB miss, the page table in memory is consulted and the mapping is loaded into the TLB.

Effective Access Time (EAT) = hit-ratio × (TLB time + memory time) + miss-ratio × (TLB time + 2 × memory time). A high hit ratio keeps EAT close to a single memory access.

Conclusion: Paging provides flexible, fragmentation-free allocation, and the TLB makes its address translation fast enough for practical use.

10 Marks Q14. What is Virtual Memory? Explain Demand Paging and the steps in handling a Page Fault.
Virtual memory is a technique that allows the execution of processes that are not completely in main memory, giving the illusion of a very large memory by using disk space as an extension of RAM.
Advantages
  • Programs larger than physical memory can run.
  • More processes fit in memory → higher degree of multiprogramming.
  • Less I/O needed to load or swap programs.
Demand Paging

A page is loaded into memory only when it is needed (referenced), not in advance. A valid–invalid bit in the page table indicates whether a page is in memory. This is a form of lazy swapping done by a pager.

Page Fault Handling Steps
  1. The CPU references a page; the page table shows the page is not in memory (invalid bit) → a page-fault trap occurs.
  2. The OS checks an internal table to confirm the reference is valid (otherwise terminate the process).
  3. A free frame is located (or one is freed by page replacement).
  4. The required page is read from disk into the free frame (disk I/O).
  5. The page table is updated, setting the valid bit.
  6. The instruction is restarted from where it faulted.

Effective Access Time = (1 − p) × memory access + p × page-fault service time, where p is the page-fault rate. Because page-fault service is very slow (disk), p must be kept extremely low.

Conclusion: Demand paging makes virtual memory practical by loading only the needed pages, but performance is highly sensitive to the page-fault rate.

10 Marks Q15. Explain Page Replacement Algorithms (FIFO, Optimal, LRU). What is Belady's Anomaly?

When a page fault occurs and no free frame is available, the OS must select a victim page to replace. The goal is to minimize the page-fault rate.

1. FIFO (First-In First-Out)

Replaces the page that has been in memory the longest. Simple to implement using a queue, but performs poorly because an old page may still be heavily used. It can suffer from Belady's Anomaly.

2. Optimal (OPT / MIN)

Replaces the page that will not be used for the longest time in the future. It gives the lowest possible page-fault rate but is impossible to implement in practice (requires future knowledge); it is used as a benchmark.

3. LRU (Least Recently Used)

Replaces the page that has not been used for the longest period of time, approximating the optimal algorithm using the recent past. Implemented with counters or a stack. It performs well but has hardware overhead.

Belady's Anomaly

Normally, increasing the number of frames decreases page faults. Belady's Anomaly is the counter-intuitive situation where, for certain reference strings, increasing the number of frames increases the number of page faults. It occurs in FIFO but not in stack-based algorithms like LRU and Optimal.

AlgorithmBasis of replacementBelady's Anomaly?
FIFOOldest pageYes
OptimalFuture useNo
LRUPast useNo

Conclusion: LRU is the most practical, closely approximating Optimal while avoiding Belady's Anomaly.

5 Marks Q16. What is Thrashing? What are its causes and how is it controlled?
Thrashing is a condition in which a process spends more time paging (swapping pages in and out) than executing, causing severe performance degradation.
Cause

It occurs when a process does not have enough frames to hold its working set (the set of pages it is actively using). High page-fault rate forces constant swapping; as the OS sees low CPU utilization it increases the degree of multiprogramming, worsening the problem.

Control Techniques
  • Working-Set Model — give each process enough frames for its working set; suspend processes if total demand exceeds available frames.
  • Page-Fault Frequency (PFF) — monitor fault rate; allocate more frames if too high, reclaim if too low.
  • Reduce the degree of multiprogramming by swapping out some processes.

In short: thrashing is prevented by ensuring each active process has enough frames for its working set.

Unit III · Deadlocks & Disk Management
10 Marks Q17. What is a Deadlock? Explain the four necessary conditions and methods for handling deadlocks.
A deadlock is a situation in which a set of processes are blocked because each process is holding a resource and waiting for another resource held by another process in the set, so none can proceed.
Four Necessary Conditions (Coffman Conditions)

A deadlock can arise only if all four hold simultaneously:

  • Mutual Exclusion — at least one resource is held in a non-sharable mode.
  • Hold and Wait — a process holds at least one resource and is waiting for additional resources held by others.
  • No Preemption — a resource can be released only voluntarily by the process holding it.
  • Circular Wait — a circular chain of processes exists, each waiting for a resource held by the next.
Methods for Handling Deadlocks
  • Deadlock Prevention — ensure at least one of the four conditions cannot hold (e.g., require all resources at once to remove hold-and-wait, or impose ordering to remove circular wait).
  • Deadlock Avoidance — use prior information about resource needs to keep the system in a safe state (e.g., Banker's Algorithm).
  • Deadlock Detection & Recovery — allow deadlocks, detect them via a wait-for graph, then recover by terminating processes or preempting resources.
  • Ignore the problem (Ostrich Algorithm) — assume deadlocks are rare; used by most general-purpose OSes like UNIX and Windows.

Conclusion: Breaking any one of the four conditions prevents deadlock; the chosen method trades off overhead against resource utilization.

10 Marks Q18. Explain the Banker's Algorithm for deadlock avoidance with the concept of a safe state.
The Banker's Algorithm is a deadlock-avoidance algorithm that tests whether granting a resource request will leave the system in a safe state; if not, the request is delayed.
Safe State

A state is safe if there exists a safe sequence of all processes <P1, P2, … Pn> such that the resources each Pi needs can be satisfied by the currently available resources plus those held by all Pj (j < i). If no such sequence exists, the state is unsafe and may lead to deadlock.

Data Structures (n processes, m resource types)
  • Available[m] — number of available instances of each resource.
  • Max[n][m] — maximum demand of each process.
  • Allocation[n][m] — currently allocated to each process.
  • Need[n][m] = Max − Allocation — remaining requirement.
Safety Algorithm
  1. Let Work = Available and Finish[i] = false for all i.
  2. Find an i where Finish[i] = false and Needi ≤ Work.
  3. If found: Work = Work + Allocationi, set Finish[i] = true, repeat step 2.
  4. If all Finish[i] = true, the system is in a safe state.
Resource-Request Algorithm

When process Pi requests resources: if Request ≤ Need and Request ≤ Available, the system pretends to allocate and runs the safety check. If the resulting state is safe, the request is granted; otherwise Pi must wait.

Conclusion: The Banker's Algorithm never enters an unsafe state, thus avoiding deadlock, but requires advance knowledge of maximum resource needs.

10 Marks Q19. Explain the various Disk Scheduling Algorithms (FCFS, SSTF, SCAN, C-SCAN, LOOK).

Disk scheduling aims to minimize seek time — the time to move the disk arm to the desired cylinder — thereby improving disk throughput.

1. FCFS (First-Come First-Served)

Requests are served in arrival order. Fair and simple, but causes large arm movement and poor performance under heavy load.

2. SSTF (Shortest Seek Time First)

Selects the request closest to the current head position. Gives better throughput than FCFS but may cause starvation of far-away requests.

3. SCAN (Elevator Algorithm)

The head moves in one direction servicing all requests until it reaches the end of the disk, then reverses direction. Like an elevator. Provides uniform wait time.

4. C-SCAN (Circular SCAN)

The head services requests in one direction; on reaching the end it jumps back to the start without servicing on the return. This gives more uniform wait times than SCAN.

5. LOOK / C-LOOK

Improvements over SCAN/C-SCAN: the arm only goes as far as the last request in each direction instead of the physical end of the disk, saving unnecessary movement.

AlgorithmStrengthWeakness
FCFSFair, simpleHigh seek time
SSTFLow seek timeStarvation
SCANNo starvationLong wait at extremes
C-SCANUniform waitExtra return sweep

Conclusion: SSTF and SCAN-family algorithms greatly outperform FCFS; SCAN/C-SCAN are common defaults as they balance performance and fairness.

Unit IV · File Systems & Security
10 Marks Q20. Explain the various File Allocation Methods (Contiguous, Linked, Indexed) with their advantages and disadvantages.

File allocation methods determine how disk blocks are assigned to files, affecting storage efficiency and access speed.

1. Contiguous Allocation

Each file occupies a set of contiguous blocks on disk. The directory stores the starting block and length.

  • Advantages: simple, supports both sequential and direct access, fast.
  • Disadvantages: external fragmentation; difficult to grow a file; finding space requires algorithms like first/best fit.
2. Linked Allocation

Each file is a linked list of disk blocks scattered anywhere; each block contains a pointer to the next. The directory stores pointers to the first and last blocks.

  • Advantages: no external fragmentation; files can grow easily.
  • Disadvantages: only efficient for sequential access; pointers consume space; reliability issue if a pointer is lost. (FAT is a variant.)
3. Indexed Allocation

All block pointers of a file are gathered into one index block. The directory points to the index block.

  • Advantages: supports direct access without external fragmentation.
  • Disadvantages: index block overhead, especially for small files; large files may need multilevel or linked index blocks.
MethodAccessFragmentation
ContiguousSequential + DirectExternal
LinkedSequential onlyNone (external)
IndexedSequential + DirectNone (external)

Conclusion: Indexed allocation (and combined schemes like UNIX inodes) is most widely used as it balances direct access with efficient space use.

5 Marks Q23. What is a Race Condition? How is it prevented?
A race condition is a situation where two or more processes access and manipulate the same shared data concurrently, and the final result depends on the order in which their instructions are interleaved.
Example

If two processes both execute counter++ (which is really load, increment, store) on a shared counter without synchronization, one update can overwrite the other, producing a wrong value.

Prevention
  • Enforce mutual exclusion over the critical section using locks, semaphores or monitors.
  • Use atomic hardware instructions (Test-and-Set, Compare-and-Swap).
  • Ensure only one process accesses shared data at a time.

In short: a race condition is solved by serialising access to shared data through proper synchronization.

10 Marks Q24. Explain the Producer–Consumer problem and its complete semaphore-based solution.
The Producer–Consumer (bounded-buffer) problem is a classic synchronization problem where a producer process creates items and places them in a shared buffer of fixed size n, while a consumer process removes them. They must be synchronized so the producer does not add to a full buffer and the consumer does not remove from an empty one.
Synchronization Requirements
  • The producer must wait if the buffer is full.
  • The consumer must wait if the buffer is empty.
  • Producer and consumer must not access the buffer simultaneously (mutual exclusion).
Semaphores Used
  • mutex = 1 — provides mutual exclusion for buffer access.
  • empty = n — counts the number of empty slots.
  • full = 0 — counts the number of filled slots.
Producer Code

do { produce item; wait(empty); wait(mutex); add item to buffer; signal(mutex); signal(full); } while(true);

Consumer Code

do { wait(full); wait(mutex); remove item from buffer; signal(mutex); signal(empty); consume item; } while(true);

How It Works

Before producing, the producer does wait(empty) — blocking if there are no empty slots. Before consuming, the consumer does wait(full) — blocking if there is nothing to consume. The mutex guarantees only one process touches the buffer at a time, preventing race conditions. The invariant empty + full = n always holds.

Conclusion: The three-semaphore scheme cleanly solves both mutual exclusion (mutex) and condition synchronization (empty, full). Note the order matters — doing wait(mutex) before wait(empty) would cause deadlock.

10 Marks Q27. Explain hardware support for synchronization — Test-and-Set and Compare-and-Swap (CAS) — and why software locks alone fail.
Hardware support provides atomic (indivisible) instructions that read and modify a memory word in a single, uninterruptible step — the foundation on which correct locks and semaphores are built.
Why a Software Lock Alone Fails

A naive lock does if (lock == 0) lock = 1; — but these are two separate steps. Two threads can both read lock == 0 before either sets it to 1, so both enter the critical section. The check and the set must be one atomic operation, which only hardware can guarantee.

1. Test-and-Set

Atomically returns the old value of a lock variable and sets it to true in one step:

boolean test_and_set(boolean *target){ boolean old = *target; *target = true; return old; }

A thread spins with while(test_and_set(&lock)); — it enters only when the old value was false. Because the read-and-set is atomic, only one thread can succeed.

2. Compare-and-Swap (CAS)

Atomically compares a variable with an expected value and updates it to a new value only if they match:

int compare_and_swap(int *v, int expected, int new){ int temp = *v; if(*v == expected) *v = new; return temp; }

A lock is acquired with while(compare_and_swap(&lock, 0, 1) != 0);. CAS is more powerful than Test-and-Set and is the basis of modern lock-free data structures and atomic variables.

Related Hardware Support
  • Memory barriers — force memory operations to complete in order, preventing CPU instruction reordering and cache-inconsistency bugs.
  • Atomic variables — variables whose operations (increment, compare) complete in one indivisible step.

Conclusion: Atomic hardware instructions like Test-and-Set and CAS make mutual exclusion correct; higher-level tools (mutexes, semaphores) are built on top of them.

10 Marks Q28. Explain IPC using Shared Memory (POSIX) and Pipes with their system calls.
Inter-Process Communication (IPC) lets cooperating processes exchange data. Two common single-host mechanisms are shared memory and pipes.
1. Shared Memory (POSIX)

A region of memory is mapped into the address space of two or more processes; they communicate by reading and writing it directly. The POSIX system calls are:

  • shm_open() — create/open a shared-memory segment (appears under /dev/shm/ in Linux).
  • ftruncate() — set the size of the segment.
  • mmap() — map the segment into the process's address space, returning a pointer for read/write.
  • munmap() — unmap the segment after use.
  • shm_unlink() — remove the segment.

Pros: fastest IPC (no kernel copy after mmap), efficient for large data. Con: provides no synchronization — a semaphore or mutex must be added manually.

2. Pipes

A pipe provides one-way (half-duplex) communication between related processes, built on kernel I/O buffers. The call int pipe(int fd[2]) opens two descriptors: fd[0] for reading and fd[1] for writing. Since a child inherits open descriptors after fork(), a parent and child can communicate through the pipe using read() and write().

Properties: the pipe inode lives in the kernel (not the file system), it exists only while its creator process lives, and it has limited capacity (65,536 bytes since Linux 2.6.11).

Conclusion: Shared memory suits high-speed bulk transfer (with separate synchronization), while pipes give simple, naturally-synchronized communication between related processes.
